## fix(dev-chat): apply diffs that create new files

- `backend/agents/dev_assist.py`: added `_normalize_create_delete_headers()` and wired it into `apply_diff()`. The dev-chat LLM frequently produces diffs of the shape

  ```
  --- a/frontend/src/layouts/MainLayout.tsx
  +++ b/frontend/src/layouts/MainLayout.tsx
  @@ -0,0 +1,N @@
  +...
  ```

  to *create* a file, but the canonical form expected by `git apply` is `--- /dev/null` for the source header. Without that, `git apply` aborts with `error: <path>: No such file or directory` and the Apply button in the dev-chat drawer surfaced the alert `Diff did not apply: error: frontend/src/layouts/MainLayout.tsx: No such file or directory`.

  The normalizer walks every `--- ` / `+++ ` file-header pair, peeks at the first `@@` hunk, and:
  - rewrites `--- a/<path>` â†’ `--- /dev/null` when the hunk header is `@@ -0,0 +N,N @@` (creation), or when the source path doesn't exist on disk under `converted/`;
  - rewrites `+++ b/<path>` â†’ `+++ /dev/null` when the hunk header is `@@ -N,N +0,0 @@` (deletion).

  Skipped when `reverse=True` because the original (already-accepted) body is what `git apply --reverse` expects.

  Net effect: dev-chat can now land patches that introduce missing files like `MainLayout.tsx`, fixing the converted-app crash `Failed to resolve import "../layouts/MainLayout"` reported from the live `vite` log.

## fix(dev-chat): apply diffs on CRLF working trees (Windows)

- `backend/agents/dev_assist.py`: added `--ignore-whitespace` to the `git apply` argv in `apply_diff()`. On Windows the converted-app files are written with CRLF line endings, but the LLM emits unified diffs with LF context lines. `git apply` compares context byte-for-byte, so even when the visible text matches exactly, the patch is rejected with `error: patch failed: <path>:<line>` / `error: <path>: patch does not apply` â€” surfaced in the dev-chat overlay as the `Diff did not apply` alert.

  Reproducer: clicking **Apply + Restart** on the `sass-embedded` diff against `uploads/60f92f3e34b5/converted/frontend/package.json` (CRLF) failed at `package.json:22`, even though line 22 in the file was an exact match for the diff context. With `--ignore-whitespace`, git matches context modulo whitespace/EOL differences and the same diff lands cleanly.

  Kept `--whitespace=nowarn` (suppresses warnings about trailing whitespace introduced by the patch itself) and `--recount` (tolerant of off-by-one hunk lengths). The new flag covers a third failure mode â€” context-line EOL drift â€” that the other two don't.

## fix(dev-chat): apply diffs that create new files (round 2 â€” body sanitization)

- `backend/agents/dev_assist.py` (`_normalize_create_delete_headers`): the previous round only fixed the `--- a/<path>` â†’ `--- /dev/null` header rewrite. That stopped the "No such file or directory" failure but unmasked a *second* git-apply failure: **"new file frontend/src/layouts/MainLayout.tsx depends on old contents"** (reproduced via the Apply button in the dev-chat drawer â€” screenshot in this thread).

  `git apply` raises that error when the patch is recognized as a creation (source = `/dev/null`) but the hunk body contains *any* line that isn't `+`-prefixed â€” i.e. context lines (leading space) or `-` lines. Creation patches are required to be additions-only. The LIGHT_MODEL frequently produces a creation diff where the indented JSX/TSX block closures (e.g. `</div>` and `}` with their leading whitespace) lose their `+` prefix and become context lines, and/or it includes a stray `-` line copied from a sibling file.

  Fix: when `is_new` is detected, walk the file block's hunks and rewrite the body so:
  - Lines starting with `+` are kept verbatim.
  - Lines starting with `-` are dropped (you cannot delete from a file that doesn't exist yet).
  - Lines starting with ` ` (context) are converted to `+` + (line without the leading space) â€” for a brand-new file every content line *is* an addition.
  - `\ No newline at end of file` markers are preserved untouched.
  - The hunk header is then re-emitted as `@@ -0,0 +1,<plus_count> @@` so the `+N` count matches the actual number of `+` lines we wrote. This is critical even with `git apply --recount`, which only tolerates *small* miscounts.

  Handles multi-hunk creation blocks (rare but the LLM does produce them when a file is "logically" split): each `@@` boundary inside a single creation block flushes the previous hunk with its own corrected header before starting the next.

  Pairs with the header-rewrite from the earlier same-day commit â€” together they make the dev-chat apply path tolerant of every realistic creation-diff shape the LLM emits.

## fix(dev-chat): multi-file creation diffs â€” emit `diff --git` separators

- `backend/agents/dev_assist.py` (`_normalize_create_delete_headers`): even after the header-rewrite + body-sanitization fixes, the **multi-file creation diff** the LLM emitted for the "create MainLayout + LoginLayout + 30 page stubs" turn was still failing in the apply path with `error: new file frontend/src/layouts/MainLayout.tsx depends on old contents`. Each individual file block (`--- /dev/null` + `+++ b/X` + the `+1,N`-counted body) was a clean creation patch in isolation, but concatenating two of them together broke git apply.

  Root cause, bisected with a standalone test: `git apply --recount` (which we pass to tolerate the small `+N` count miscounts the LLM produces) keeps reading hunk-body lines past the header-declared count until it finds a "diff" boundary. The next block's `--- /dev/null` line *starts with* `-`, so git's parser consumed it as additional deletion-context for the previous hunk â€” and a creation patch (source = `/dev/null`) with any deletion-shaped lines triggers exactly the `depends on old contents` error. Confirmed by removing `--recount`: the multi-file diff applied cleanly. Confirmed again by adding `diff --git` separators between blocks: applied cleanly even *with* `--recount`.

  Fix: in the `is_new` branch, prepend a canonical pair of headers before the `--- /dev/null` line â€” `diff --git a/<path> b/<path>` and `new file mode 100644` â€” which is the exact form `git diff` itself emits for newly-created files. With those present, git's apply parser sees an unambiguous boundary between blocks and stops trying to read further body lines. A small guard checks the last 3 emitted lines to avoid double-emission if the LLM already produced a `diff --git` header.

  Verified end-to-end against the exact diff from the 3:01 AM Apply-button failure: the 4-block test that previously returned `rc=128 / depends on old contents` now returns `rc=0` (apply succeeds). The same fix works for both `--recount`-on and `--recount`-off code paths, so the existing `--recount` flag is kept as belt-and-suspenders for non-creation diffs with mis-counted hunks.

  Pairs with the earlier same-day fixes: header rewrite (round 1), body sanitization for stray context / deletion lines (round 2), and now block-boundary headers (round 3). Together these make the dev-chat Apply path robust to every realistic shape of LLM-emitted creation diff the demo apps have produced so far.

## fix(dev-chat): skip creation diffs that case-collide with existing files (round 4)

  In `_normalize_create_delete_headers`, when the `is_new` branch detects a case-insensitive collision, it now DROPS the entire file block from the normalized output instead of emitting it â€” git never sees the bogus creation, so the rest of the multi-file diff can land.   Pairs with new prompt rule 9 (below) â€” the rule reduces the rate of case-colliding creation diffs at the source, the normalizer catches the ones the LLM still emits.

## fix(dev-chat): stop the converted run BEFORE applying the diff, not after

- `backend/main.py` (`dev_chat_apply_diff` endpoint): the Apply + Restart flow used to call `apply_diff()` first and `restart_run()` afterwards. On Windows that's a race: while git is writing the patch, the running converted app still has file handles open â€” vite watches the whole `frontend/src/` subtree and re-bundles on every change, dotnet keeps every loaded `.dll` mapped into memory, and either of those will either (a) lock the target file with EACCES so `git apply` fails mid-patch, or (b) trigger a partial recompile against a half-applied tree that crashes the dev server before the rest of the patch lands.

  Reordered: when `restart=True` (and not a dry-run), the endpoint now (1) finds the active run for the session, (2) calls `stop_run(active.run_id)` which kills the parent .bat AND sweeps `run.port` + `run._backend_port` for surviving listeners (the same-day `stop_run` fix above is what makes this reliable on multistack launchers), (3) sleeps 0.5s for the OS to fully release the TCP binding, (4) THEN runs `apply_diff` against the quiescent tree, (5) THEN calls `start_run` against the same session + converted_dir to bring up a fresh process bound to the patched code. The pre-stop snapshot of `(session_id, cwd)` is what we pass to the post-apply `start_run`, so the new run is guaranteed to land in the same workspace even if `_session_converted` mappings drift.

  Rollback path: if `apply_diff` returns `ok=False` after we already killed the previous run, the endpoint now calls `start_run` again to bring the original (unpatched) process back up. Without this the reviewer would be left staring at a dead converted app on top of a failed patch â€” broken state masquerading as a successful action. With it, a failed Apply leaves the system in the same state it found, modulo the brief downtime of the restart.

  Aligns with the user request â€” every run-converted + dev-chat pair now follows the same ordered handshake: stop the current run â†’ quiesce â†’ land the change â†’ bring up a fresh run. Works equally for the current session's converted/ folder and for any future session's converted/ folder spun up by the codegen pipeline.

## fix(scaffold,prompts): converted React app must render with production-grade UI on first `npm run dev`

### 1. `backend/agents/scaffold.py` â€” npm dep scanner is now SCSS-aware

`_scan_npm_imports()` previously grepped only `*.ts`, `*.tsx`, `*.js`, `*.jsx` for ES `import` / CJS `require` specifiers. The agent-emitted `src/styles/global/index.scss` does `@import 'bootstrap/scss/bootstrap'` â€” a real npm dep â€” but the scanner skipped it, so `package.json` shipped without `bootstrap` and Vite served zero CSS at `npm run dev`. The page then rendered with browser defaults (serif font, default blue underlined links, bullet lists) â€” exactly the screenshot.

Changes:
- Extended `_scan_npm_imports()` to also walk `*.scss`, `*.sass`, `*.less`, `*.css` and match `@use 'pkg'`, `@forward 'pkg'`, `@import 'pkg/sub'`, `@import url('pkg/sub')`. SCSS specs without a `/` (e.g. `@import 'variables'`) are treated as local partials and filtered out so we don't pollute `dependencies` with `variables`, `global`, `navbar.module`, etc. Scoped (`@scope/pkg`) and subpath-qualified (`pkg/sub`) names are kept.
- Added a `_has_scss_sources()` helper that forces `sass-embedded` into devDependencies whenever ANY `.scss` / `.sass` file ships â€” Vite refuses to compile SCSS without a sass implementation installed, and that omission is a silent footgun.
- Pinned versions added to `_NPM_DEP_VERSIONS`: `bootstrap@^5.3.3`, `react-bootstrap@^2.10.5`, `bootstrap-icons@^1.11.3`, `@fortawesome/fontawesome-free@^6.6.0`, `jquery@^3.7.1`. Pinned `sass-embedded@^1.79.4` added to `_NPM_DEV_VERSIONS`. Without pins the fallback path uses `latest`, which has bitten us with breaking v6 majors on legacy v3/v4-styled apps.

### 2. `backend/agents/scaffold.py` â€” `_ensure_react_manifests()` now AUGMENTS an existing `package.json`

Previous behaviour: the function bailed at line 1 if a `package.json` already existed. That meant whenever the LLM emitted a `package.json` (which it does for every conversion), the scaffold's safety-net dep list was silently discarded â€” including the SCSS-derived deps from change (1) above. Net effect: the new SCSS scanner would find `bootstrap` but the scaffold would never write it.

Now:
- When `package.json` exists, parse it, merge any newly-scanned deps that aren't already in `dependencies` OR `devDependencies` (don't clobber agent-chosen versions), sort, and write back. Logged as `package.json (augmented: +bootstrap,sass-embedded,...)` so the diff is auditable.
- Delete the sibling `package-lock.json` when we augmented â€” a stale lockfile under `node_modules` otherwise satisfies Vite without the new deps and the install effectively no-ops.
- The Vite/TS config + `index.html` backfill still runs whether or not we augmented, so half-emitted projects don't get stuck.

Smoke-tested against `uploads/378d02ddd74c/converted/frontend`: the augmented manifest now lists `"bootstrap": "^5.3.3"` under `dependencies` and the existing `sass-embedded` dev dep is preserved.

### 3. `backend/agents/scaffold.py` â€” new `_copy_static_assets()` step

Code-generation never copies binaries â€” it only writes text. So the legacy `<img src="../assets/images/logo-dashboard.png">` references that the LLM faithfully ports from the source template hit 404 on every route. Added a programmatic step wired into `ensure_build_manifests()` that mirrors legacy static-asset folders into `converted/frontend/public/assets/<bucket>/`.

Implementation:
- `_STATIC_ASSET_DIRS` enumerates the typical legacy locations (`src/images`, `src/img`, `src/assets`, `src/fonts`, `public`, `wwwroot/images`, `Content/images`, `static`, `assets`, â€¦). The walker tries each, both at `source_root` and one level under (`source/<ProjectName>/...`).
- `_STATIC_ASSET_GLOBS` covers `*.png|jpg|jpeg|gif|svg|webp|ico|woff|woff2|ttf|eot|otf|mp4|webm`. Files land under `public/assets/<bucket>/` where `<bucket>` is the source folder name lowercased (so legacy `src/images/foo.png` â†’ `public/assets/images/foo.png`, matching the absolute `/assets/images/foo.png` URL the codegen prompts now mandate â€” see change (4)).

### P1. System.Text.Json request-DTO null tolerance

- **FK id fields MUST be `int?`** (legacy sends `null` for unset FKs).
- **Any sanitizable string field MUST be `string?`** â€” default to `string?` for ALL request-DTO strings when in doubt.
- **Response-DTO side gets the opposite default** (`string` if entity is non-nullable) â€” response DTOs are stricter because the server controls them.

### P4. Print pipeline â€” `display:none` vs `@media print`
