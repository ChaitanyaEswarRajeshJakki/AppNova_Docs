# 2026-06-11 13:20 — Fix browser-test `port_never_bound` (localhost vs 127.0.0.1 on Windows)

Symptom: Browser Test fails with `port_never_bound` — "Dev server never came up at
http://localhost:5050 (TCP poll to localhost:5050 timed out after 30s)" — even though
the app is up and reachable at http://127.0.0.1:5050.

Root cause: a harness-wide host inconsistency.
- Dev servers bind the IPv4 loopback: Vite via `--host 127.0.0.1` / `HOST=127.0.0.1`;
  the converted frontend's vite.config uses `host: '127.0.0.1'`. (Kestrel bound to
  `localhost` dual-binds, so 127.0.0.1 reaches the backend too.)
- But the run-manager reported the base URL as `http://localhost:{port}` and
  browser_test derived its TCP-poll host from that URL.
- On Windows, `localhost` resolves to IPv6 `::1` first — where nothing listens — so
  the poll (and a browser typing `localhost`) times out, while `127.0.0.1` works.

Fix — standardize the harness poll/URL on 127.0.0.1 (the address every server
actually answers on):
- `backend/agents/pipeline/post_codegen_browser_loop.py`: browser_test `base_url`
  localhost → `127.0.0.1` (poll + Playwright navigation + error text).
- `backend/agents/tools/run_manager.py`:
  - RunState `url` (the "Open in tab" link shown to the user) localhost → 127.0.0.1.
  - `wait_for_http_ready` default `host` localhost → 127.0.0.1 (matches
    `wait_for_port`, which already defaulted to 127.0.0.1).
  - Left the dotnet-candidate `ASPNETCORE_URLS=http://localhost:{port}` as-is —
    Kestrel dual-binds `localhost`, so 127.0.0.1 reaches it and localhost still
    works for dotnet-only apps.

Why no regression: 127.0.0.1 is the lowest common denominator — Vite/Node bind it,
run-manager forces it (`--host 127.0.0.1`, `HOST=127.0.0.1`), and Kestrel-on-localhost
includes it. `localhost` was NOT reliably bound by Node on Windows. Applies to all
future conversions automatically; the converted app needs no change (it already
answers on 127.0.0.1). `py_compile` clean.

Note for users: AppNova now shows/links http://127.0.0.1:{port}. Typing `localhost`
in a browser may still fail for a Vite (Node) frontend because Node binds only the
IPv4 loopback; use the 127.0.0.1 link AppNova provides.
