# 2026-06-05 16:00 — Run manager: detached log no longer reports backend status as frontend's

## Problem
Session `378d02ddd74c` logged:
```
[detached] launcher exited rc=0; server reachable (frontend http://localhost:5050/ → 404); running in detached child windows
```
The `→ 404` was the **backend's** root status (a JWT-`[Authorize]` API that
returns 404 on `/`), not the frontend's. The previous code did:
```python
http_status = fe_status if fe_ok else be_status
probed_path = fe_path if fe_ok else be_path
```
When Vite hadn't finished its cold start within the probe window, `fe_ok=False`
caused it to borrow `be_status=404` and print it as if the frontend had answered.

## Fix — `backend/agents/tools/run_manager.py`

### Multistack detached branch (`_launch_and_await_ready`)

| Change | Detail |
|---|---|
| `fe_confirmed` tracked separately | `fe_confirmed = fe_ok` — decoupled from backend verdict |
| No status substitution | If `fe_confirmed=False`, `http_status=None`, `probed_path=None` |
| TCP-only backend clears status | `be_status, be_path = None, None` (was `None, "(tcp)"`) so it can't leak into log |
| `be_status_for_log` | Captures backend's own HTTP status for use in the log message only |
| Frontend probe timeout | `12.0s → 10.0s` (per spec: "bumped 6s → 10s") |
| Context-aware log message | Three cases: fe confirmed → `frontend …/ → 200`; backend-only → `backend :5051 confirmed (→ 404); frontend :5050 still starting`; single-stack unchanged |
| `frontend_ready` in SSE | Both `phase` and `ready` events now carry `frontend_ready: bool` |

### Logic trace for `378d02ddd74c`
- Backend answers `:5051` with HTTP 404 → `be_ok=True, be_status_for_log=404`
- If Vite answers within 10s → `fe_confirmed=True`, log: `frontend …/ → 200`
- If Vite still starting → `fe_confirmed=False`, log: `backend :5051 confirmed (→ 404); frontend :5050 still starting in its window`
- Either way, no phantom "frontend → 404"
