# Changes — 21-05-2026

## 1. Reactive multipass escalation now reachable when attempt 1 emits no JSON

**Bug:** session `e330995920ce` (2026-05-20 13:44) ran migration-planner on a
340-source-file project. Attempt 1 produced no parseable `file_map.json`, the
supervisor dispatched a full repair pass, and the repair pass hit the 1800 s
agent timeout (`[FileMap] repair pass failed to run: Timed out after 1800
seconds`). `file_map.json` was never written, migration-planner was downgraded
to `status=error`, and `code-generation` / `code-review` / `documentation` /
`testing` / `ui-ux` were all SKIPPED — the whole conversion produced nothing.

**Root cause:** the reactive section-per-pass escalation in `_run_one`
([backend/agents/supervisor.py:1446](backend/agents/supervisor.py#L1446)) is
gated on `source_files_estimate >= 200`, but `source_files_estimate` was
derived *only* from the planner's own emitted `extracted["meta"]
["source_files_total"]`. When attempt 1 fails to emit any parseable JSON —
the exact large-project failure mode — `extracted is None`, the estimate
stays `0`, and the `>=200` gate can never fire. The multipass runner (the one
tool built for projects too big for a single turn) was therefore unreachable
precisely when it was needed, leaving only the doomed single-pass + repair
path that times out.

**Fix:** [backend/agents/supervisor.py:1312-1325](backend/agents/supervisor.py#L1312-L1325)
— when `source_files_estimate` is still `0` after reading the planner meta,
fall back to `_estimate_source_file_count(project_dir)`, the same on-disk
heuristic the proactive early-multipass gate already uses. A 340-file project
now reports ~268 (the heuristic counts `.php`/`.js`/`.html`; `.scss`/`.css`
are excluded by design), which clears both the `>120` targeted-completion
gate and the `>=200` multipass-escalation gate.

**Risk:** minimal — the new branch only runs when the estimate would
otherwise be `0`, and `_estimate_source_file_count` returns `0` on any error,
so the worst case is identical to the old behaviour. The proactive
early-multipass path (threshold `1`) still runs first for every project; this
patch hardens the single-pass + repair *fallback* that runs only when the
proactive multipass itself fails to return a usable result.

to the desktop checkout along with this changelog.

## 2. Migration-planner timeout now scales with project size

**Context:** the 2026-05-20 13:44 failure (session `e330995920ce`, see §1)
was a wall-clock timeout — the migration-planner repair pass ran the full
flat `1800 s` / 30-min budget and was killed mid-stream. `1800 s` is fine
for a normal repo but tight for a 340-source-file one, where both the
single-pass + repair fallback and each multipass section's continuation
loop legitimately take longer.

**Change A — size-scaled timeout
([backend/agents/supervisor.py](backend/agents/supervisor.py)):**

- New `_scaled_planner_timeout(base, project_dir)` helper + two env-tunable
  constants `_PLANNER_TIMEOUT_PER_FILE` (default `8 s`) and
  `_PLANNER_TIMEOUT_CAP` (default `5400 s` / 90 min), placed next to
  `_estimate_source_file_count`.
- For repos with ≤200 source files the planner timeout is unchanged
  (`1800 s`). Above that, it adds 8 s per file, hard-capped at 90 min — so
  a wedged subprocess still dies rather than hanging for hours. A 340-file
  repo now gets `1800 + 140*8 = 2920 s` (~49 min).
- `_run_one`'s `_invoke` closure applies the scaled budget for
  `agent_id == "migration-planner"` only. Because the multipass runner
  drives every section through this same `_invoke`, each section also gets
  the larger budget — directly answering "even a slow multipass section
  has room."
- Tunable via `APPNOVA_PLANNER_TIMEOUT_PER_FILE` /
  `APPNOVA_PLANNER_TIMEOUT_CAP`; a heuristic miss degrades to the old
  `1800 s`, so correctness is never on the line.

**Change B — output-token ceiling: no change needed.**
`CLAUDE_CODE_MAX_OUTPUT_TOKENS` is already raised from the CLI's `32000`
default to `64000` at [backend/agents/runner.py:65](backend/agents/runner.py#L65)
and pinned on every subprocess at
[backend/agents/runner.py:586](backend/agents/runner.py#L586). `64000` is
the hard output ceiling for Sonnet 4.x (the model migration-planner runs
on); raising it further would make the API reject the request with a 400,
so it is left at `64000`.

## 4. Stopped agent narration leaking into rendered reports

**Problem:** agent cards were showing the model's between-tool-call
narration — "let me read the controllers", "now I'll grep the routes",
console reads, etc. — mixed into the final report. The report should be
*only* the report; the work log belongs in the backend run dump, not on
the card.

**Root cause — two paths:**

1. **Normal path — the `preferStream` heuristic.**
   [completeCard()](frontend/app.js) picked the report body with a
   `preferStream` rule: when the live-streamed transcript (`assistantAccum`)
   was ≥1.5× longer than the agent's clean final result (`ev.result`), it
   rendered the *whole streamed transcript* as the report. But
   `assistantAccum` is every streamed text delta concatenated — i.e. all
   the narration. The heuristic fired *exactly* when the agent narrated a
   lot, so the more the model talked, the more likely the dirty path won.
   `stripReportPreamble()` only trims the intro, so narration interleaved
   later survived.
2. **Partial / timeout path.** On a timeout, runner.py's
   `_extract_partial_assistant_text()` rebuilds a report by joining *every*
   streamed `assistant` text fragment — narration included — and that
   became `result`. `scrub_report()` only removes single filler
   *sentences*, not a whole interleaved narration block.

**Fix:**

- **[frontend/app.js](frontend/app.js)** — removed the `preferStream`
  heuristic (and its `streamLen` / `resultLen` locals). `completeCard()`
  now always prefers `ev.result` (the SDK's final answer = report only,
  already `scrub_report`'d server-side); `assistantAccum` is used *only*
  as a fallback when `ev.result` is empty. Same change applied to the
  partial branch's `partialFinal` selection.
- **[backend/agents/report_scrubber.py](backend/agents/report_scrubber.py)**
  — new `strip_to_report_body()`: cuts everything before the first
  markdown heading (a report-shaped agent always opens its deliverable
  with a heading; on a partial capture everything before it is
  narration). Conservative — if there is no heading at all (agent killed
  before reaching the report body), returns the text untouched rather
  than destroy the only rescued content. Deliberately **not** part of
  `scrub_report()`: the happy path's `result` has no narration preamble
  and many reports open with a sentence before their first heading.
- **[backend/agents/runner.py](backend/agents/runner.py)** — the
  partial-recovery path now runs `strip_to_report_body(scrub_report(...))`
  before prefixing the ⚠ timeout banner, so a rescued partial reads as a
  report, not a transcript dump.

**Risk:** low. The clean source (`ev.result`) already existed and is
already scrubbed server-side; this just stops the frontend from
overriding it with the raw transcript. Tiny behaviour change: if a repair
pass returns a shorter-but-complete report, the shorter (clean) one now
shows instead of the longer (dirty) streamed view — which is the intended
outcome. `strip_to_report_body()` smoke-tested on four cases (narration
before heading, already-clean, no-heading, empty).

**Sync:** [frontend/app.js](frontend/app.js),
[backend/agents/report_scrubber.py](backend/agents/report_scrubber.py)
and [backend/agents/runner.py](backend/agents/runner.py) copied to the
desktop checkout along with this changelog.

## 5. Migration-planner repair chip no longer stuck on "↻ repairing"

**Bug:** in a session where migration-planner's first draft was rejected
(thin body / missing `file_map.json`), the agent card painted a yellow
`↻ repairing` chip and never cleared it — the chip stayed frozen on
"repairing" even after the agent finished and the pipeline moved on to
code-generation. Observed on session `50bf87039dce` (2026-05-21 07:40):
migration-planner completed, the repair pass ran once, `file_map.json`
was extracted (333 mappings), wave 3 started — yet the card still read
"repairing".

**Root cause:** the frontend's `paintRepairChip()`
([frontend/app.js:2530](frontend/app.js#L2530)) is called with
`final=false` (live yellow "↻ repairing") on the `agent_event` SSE
message, and only flips to a final state — `↻ repaired` / `✓ blueprint`
/ `⚠ incomplete` — when an `agent_complete` / `agent_error` event
arrives **carrying `repair_status`** (the `if (ev.repair_status)` guard
at [frontend/app.js:2307](frontend/app.js#L2307)). But the supervisor
stored `repair_status` only on its internal `result` object
([backend/agents/supervisor.py:1594](backend/agents/supervisor.py#L1594))
and never copied it onto the `agent_complete` / `agent_error`
`event_payload` — that payload forwarded `file_coverage`,
`ui_inventory_shape`, `deploy_audit` and `context_attestation`, but not
`repair_status`. So the completion event had no `repair_status`, the
guard was always false, `paintRepairChip(final=true)` never ran, and the
chip stayed on whatever the last `final=false` paint left it.

**Fix:** [backend/agents/supervisor.py](backend/agents/supervisor.py) —
forward `repair_status` onto the completion event alongside the other
diagnostic blobs (`if result.get("repair_status"):
event_payload["repair_status"] = result["repair_status"]`). The
`agent_complete` / `agent_error` event now carries it, the frontend
repaints the chip to its true final state on completion.

**Risk:** none — purely additive. Only migration-planner ever sets
`repair_status`, the guard skips every other agent, and the frontend
already had the `final=true` rendering branch wired and waiting for the
field.

copied to the desktop checkout along with this changelog.

## 6. Deterministic zero-drop backfill for file_map.json

**Problem:** the migration-planner hand-writes the `mappings[]` list in
`file_map.json` — one row per source file. On large projects (300+
files) it routinely drops 8–15 rows mid-list: an enumeration slip, not a
decision. Those source files then have no row, so code-generation is
never told to convert them — they vanish from the output silently. The
existing validator (`validate_planner_quality`) detects the gap and
shouts `file_map.json is missing N of M source files`, but nothing
*fixed* it: code-review only judges files that were generated, and the
coverage audit's alarm floor is 70%, so a ~96%-covered map (8–15 missing
of 345) sails through green with the hole intact. The repair / multipass
retries are themselves LLM passes and can drop files again — observed on
session `50bf87039dce` (2026-05-21), where the repair pass ran and still
left 8 files missing.

**Fix — pure set arithmetic, no LLM:**

- **New `backfill_missing_source_rows()`
  ([backend/agents/planner_polish.py](backend/agents/planner_polish.py)).**
  `source_inventory.json` is the machine-generated ground truth (every
  real source file, no slip). `inventory − mapped_sources` is the exact
  set of dropped files. For each, the function appends a `kind="SKIPPED"`
  row with empty `targets[]`, `_backfilled: true`, and a `skip_reason`
  recording the provenance. It also bumps `meta.skipped_count` /
  `meta._backfilled_count` so coverage maths stay honest. A deterministic
  backfill running *before* code-generation has no converted tree to
  stem-match against, so it cannot safely invent a cross-stack target
  path — `SKIPPED` is the honest classification; the `_backfilled` marker
  flags the row for human target pairing in the Review UI. Subtraction
  cannot itself drop a file, so this is a hard zero-drop guarantee.
- **Wired into `_run_one`
  ([backend/agents/supervisor.py](backend/agents/supervisor.py)).** Runs
  at the single convergence point where `file_map.json` is persisted —
  after the initial pass, repair, and multipass have all converged — so
  it catches drops from every path. Backfilled paths are recorded on
  `repair_status` (`backfilled_sources`, `backfilled_count`), and the now
  factually-resolved `is missing` body-failure strings are filtered out
  of `repair_status.body_failures` so the §5 repair chip doesn't paint a
  permanent red "incomplete" for a map that is now structurally whole.

**Effect:** `file_map.json` always carries one row per real source file.
The `missing N of M` validator failure can no longer fire. Dropped files
are no longer silent — they appear in the map (and Review UI) flagged
`_backfilled` for human pairing.

**Honest limitation:** backfilled rows are `SKIPPED`, not auto-converted
— a reliable cross-stack target cannot be derived deterministically
before code-generation runs. This is a safety floor (nothing is lost,
everything is visible), not a substitute for the planner mapping the
file correctly. A future targeted micro-pass (map *only* the dropped
files) could convert them; deliberately out of scope here to keep the
change deterministic and LLM-free.

**Risk:** low — additive. If `source_inventory.json` is missing /
unreadable the function returns `[]` and behaviour is unchanged; the
whole call is wrapped in a non-fatal try/except.

## 7. Code-generation chunk parallelism + adaptive usage-window safeguard

**Context:** `run_codegen_multipass` split the planner's `mappings[]`
into chunks and ran them **strictly sequentially** (`for chunk in
chunks:`), one `claude -p` turn at a time with a 30 s cooldown between.
On a 345-file project that is ~7 chunks × ~20 min ≈ 2.5 h wall clock.
The sequential design and its cooldown were built around a per-minute
TPM (tokens-per-minute) ceiling — but this deployment authenticates the
CLI with a **Claude subscription** (OAuth, `claudeAiOauth` in
`~/.claude/.credentials.json`), **not** a pay-per-token API key. A
subscription has **no per-minute TPM wall**; its limiter is the rolling
**5-hour usage window**. So the sequential gate was guarding against a
wall that does not exist for this deployment.

**Change A — wave-based bounded parallelism
([backend/agents/codegen_multipass.py](backend/agents/codegen_multipass.py)).**
The per-chunk body was extracted into a pure `_process_chunk()`
coroutine (no shared-counter mutation — it returns an outcome dict). The
main loop now runs chunks in **waves of `parallelism`**: it fires N
chunks with `asyncio.gather`, waits for the wave, aggregates outcomes,
cools down, fires the next wave. Each chunk owns a disjoint slice of
`file_map` rows (disjoint `targets[]`), so concurrent passes write
different files and never collide. `parallelism=1` reproduces the legacy
strictly-sequential behaviour byte-for-byte.

**Change B — adaptive circuit breaker (the usage-window safeguard).**
The honest risk of going faster: a big run drains the 5-hour window
faster. The breaker addresses it directly — the first time *any* chunk
comes back rate-limited / usage-limited (`_looks_rate_limited()` matches
the same shapes as `runner._RATE_LIMIT_PATTERNS`), every remaining wave
drops to **width 1**. The run self-throttles toward the window instead
of firing N passes straight into it. Combined with the pre-existing
per-chunk `.done` resume markers, a run that *does* exhaust the window
fails cleanly and **resumes for free** on the next attempt (completed
chunks are skipped) — it never loses work, never silently half-builds,
and the "next job waits for the reset" failure mode becomes "this job
pauses and continues." A `{"phase": "throttle"}` SSE event fires when
the breaker trips, and chunk-failure/error logic now propagates
`asyncio.CancelledError` (user stop) and treats `gather` exceptions as a
single failed chunk rather than aborting the wave.

**Change C — tuning ([.env](.env),
[backend/agents/supervisor.py](backend/agents/supervisor.py)).**

- `APPNOVA_CODEGEN_PARALLELISM` — new, default `3`, clamped `[1, 6]`.
  Threaded supervisor → `run_codegen_multipass`; surfaced in the
  dispatch log line, the SSE start message, and `_codegen_multipass_meta`.
- `APPNOVA_CODEGEN_CHUNK_SIZE` `50 → 65` — with parallelism, fewer but
  slightly larger chunks mean fewer inter-wave cooldowns; 65 stays clear
  of the documented 80-row mid-mapping truncation ceiling.
- `APPNOVA_CODEGEN_COOLDOWN_SECONDS` `30 → 15` — per user note. The
  cooldown is cache-warmth + a beat for the usage window, not a TPM
  refill; 15 s keeps the prompt cache warm with less idle time.

**Expected effect:** a 345-file conversion drops from ~2.5 h toward
~50–60 min at `parallelism=3`, with no model downgrade (still Sonnet).

**Option 5 (parallel migration-planner sections) — not done, and
why.** `planner_multipass._SECTIONS` was reduced to a **single** entry
(A.4) in the 2026-04-23 cleanup — A.0/A.1/A.2/A.3/A.5/B were deleted
because nothing downstream consumed them. With one section there is
nothing to parallelize; the planner's remaining cost is one A.4 pass
plus inherently-sequential continuation calls. `planner_multipass.py`
was therefore left untouched.

**Risk:** medium — this changes a concurrency-sensitive write path.
Mitigations: (a) `parallelism=1` fully restores the old behaviour and is
one env var away; (b) chunks write disjoint file sets so concurrent
writers do not collide; (c) `run_codegen_multipass` is already wrapped
in the supervisor's try/except with a single-pass fallback; (d) `.done`
markers make every run resumable; (e) the circuit breaker bounds the
worst case to "degrades to sequential." Not run end-to-end in this
environment — recommend a first live run at `APPNOVA_CODEGEN_PARALLELISM=2`
to watch the wave/throttle logs before moving to 3.

**Sync:** [backend/agents/planner_polish.py](backend/agents/planner_polish.py),
[backend/agents/supervisor.py](backend/agents/supervisor.py),
[backend/agents/codegen_multipass.py](backend/agents/codegen_multipass.py)
and [.env](.env) copied to the desktop checkout along with this changelog.

## 8. "Socket connection was closed" now classified transient (retried)

**Bug:** session `50bf87039dce`, code-generation **chunk 5** failed at
2026-05-21 09:26 with:

```text
API Error: The socket connection was closed unexpectedly.
```

That chunk had run **2451.9 s (≈41 min) over 161 turns** — an unusually
heavy slab. A streaming HTTP turn held open that long outlives the
socket: an intermediate proxy, the server's own connection-duration
cap, or a NAT idle timeout closes the TCP connection mid-response. This
is a textbook *transient* — a retry replays against warm cache for
pennies and almost always succeeds.

**Root cause:** the runner classifies an error as retryable by matching
its text against `_TRANSIENT_PATTERNS`
([backend/agents/runner.py](backend/agents/runner.py)). That list
carried `"socket hang up"`, `"connection reset"`, `"connection error"`,
`"fetch failed"`, `"econnreset"`, `"etimedout"` … but **not** the
wording the CLI actually emitted — `"the socket connection was closed
unexpectedly"`. So `_is_transient_error()` returned False, the runner
logged `transient=False`, the retry loop hit `if not transient: break`,
and the chunk failed **without a single retry** — `codegen_multipass`
counted it `chunks_failed += 1` and moved on, leaving chunk 5 with no
`.done` marker.

**Fix:** added `"socket connection was closed"` and `"closed
unexpectedly"` to `_TRANSIENT_PATTERNS`. The error is now classified
transient → the runner retries it (longer cache-warm backoff path) →
cheap fast recovery instead of a dead chunk. Not a rate-limit pattern,
so it correctly does **not** trip the codegen circuit breaker (§7) —
a network blip is not a usage-limit.

**Note on chunk size:** §7 raised `APPNOVA_CODEGEN_CHUNK_SIZE` 50 → 65.
A bigger chunk = a longer single streaming turn = *more* exposure to
this exact socket-drop. The retry-pattern fix is the real mitigation;
if this failure mode recurs often, drop the chunk size back toward 50
to shorten each turn.

**Risk:** minimal — additive string entries to a pattern tuple. Worst
case a non-transient error sharing this wording gets one wasted retry
against warm cache (cheap). No behaviour change for any other error.

**Sync:** [backend/agents/runner.py](backend/agents/runner.py) copied to
the desktop checkout along with this changelog.

## 9. Broke the code-generation ⇄ documentation dependency cycle

**Bug:** every full pipeline run since 2026-05-20 logged

```text
[Supervisor] dependency cycle detected among
  ['documentation', 'code-generation', 'code-review', 'testing', 'ui-ux']
```

and then planned a wave `[code-generation, code-review, documentation,
testing, ui-ux]` — all five in **one** wave. `code-generation` ran;
`code-review` / `documentation` / `testing` / `ui-ux` were all **SKIPPED**
with `Required upstream agent(s) unavailable: code-generation`. Four
agents' worth of output (review, docs, test strategy, UI/UX) was lost on
every run; the only recovery was a manual re-run of just those four.

**Root cause:** each `AgentSpec` carries two dependency fields —
`upstream` (drives `compute_waves`' topological layering) and
`required_upstream` (the skip gate). They were inconsistent:

- `code-generation.upstream` listed `"documentation"`
  ([backend/config.py](backend/config.py)) — added when "every
  analytical agent" was wired into code-generation's context.
- `documentation.upstream` lists `"code-generation"` — correct, it
  documents the generated tree.

That is a `code-generation ⇄ documentation` cycle. `compute_waves`'
layered topological sort cannot resolve a cycle: with no agent ever
"ready" it bails ([backend/agents/supervisor.py:573-578](backend/agents/supervisor.py#L573-L578))
and dumps every remaining agent into one final wave. `asyncio.gather`
then launches all five at once; the four with `code-generation` in
`required_upstream` check at launch, find no code-generation output yet
(it is running concurrently beside them), and are skipped.

**Fix:** removed `"documentation"` from `code-generation.upstream`.
`documentation` is a *post*-code-generation agent — it never belonged in
code-generation's upstream. The cycle is gone; `compute_waves` now
layers the tail correctly:

```text
… analysis … → [migration-planner] → [code-generation]
   → [code-review, documentation, ui-ux] → [testing]
```

No agent is skipped. `documentation` still consumes code-generation's
output — its own `upstream` / `required_upstream` keep `code-generation`,
which is the correct direction.

**Risk:** low — removes one wrong edge from the dependency graph. The
only behaviour change is the intended one: the four post-codegen agents
now run *after* code-generation instead of being skipped. code-generation
still receives every genuine analytical upstream report.

**Sync:** [backend/config.py](backend/config.py) copied to the desktop
checkout along with this changelog.

## 10. Review diff viewer — source pane no longer renders empty

**Bug:** in the Hub → Review & Approval diff viewer, the **converted**
file rendered but the **source** file pane was blank for session
`50bf87039dce` (and any session shaped like it).

**Root cause — two compounding faults:**

1. **Wrong file_map.json.** The pipeline (supervisor + code-generation)
   writes the authoritative `file_map.json` under
   `uploads/<sid>/source/context/`. But `_load_file_map_for_review`
   ([backend/main.py](backend/main.py)) looked under
   `_session_folder(sid)/context/` — i.e. `uploads/<sid>/context/`,
   missing the `source/` segment. Finding nothing there once, it ran the
   disk-walking synthesizer and **persisted a 701-row stem-matched guess**
   to `uploads/<sid>/context/file_map.json`. Every later Review call then
   read that stale synthesized map instead of the planner's real 340-row
   map. Confirmed on disk: `context/file_map.json` = 701 rows,
   `_synthesized:true`, 493 sources prefixed `TotalBookingAI-Input/…`;
   `source/context/file_map.json` = 340 rows, clean `app/…` sources.
2. **Double wrapper prefix.** The synthesized map's source paths carry
   the input wrapper folder (`TotalBookingAI-Input/app/Helpers/Foo.php`).
   The `/api/review/<sid>/file` endpoint joins `source_root` (which is
   already `uploads/<sid>/source/TotalBookingAI-Input`) + that path →
   `…/TotalBookingAI-Input/TotalBookingAI-Input/app/Helpers/Foo.php` →
   no such file → `_read_file_safely` returns `file_missing` → the
   source pane renders empty. The converted side joins cleanly, so the
   target pane renders — exactly the reported asymmetry.

**Fix (both in [backend/main.py](backend/main.py)):**

- **Part A — `_load_file_map_for_review`** now resolves `file_map.json`
  from a priority list: `source/context/file_map.json` (the authoritative
  planner artifact) first, then the legacy `context/file_map.json`, then
  on-the-fly synthesis. The Review panel now reads the real 340-row map
  with clean `app/…` source paths that join correctly.
- **Part B — `review_get_file`** gains a wrapper-prefix seatbelt: when
  the source join returns `file_missing` AND the source path starts with
  `source_root`'s own basename, it retries once with that segment peeled
  off. This makes any prefixed path — synthesized fallback, disk-inferred
  union rows — resolve, so the empty-source-pane class of bug cannot
  silently recur.

**Note:** the stale `uploads/<sid>/context/file_map.json` (701
synthesized rows) stays on disk — harmless once Part A stops reading it
in preference to the real map. Not deleted (non-destructive); a later
cleanup could prune `_synthesized` maps once a real one exists.

**Risk:** low — Part A only changes *which* existing file is read
(prefers the more-authoritative one); Part B is a fallback that runs
*only* on an otherwise-failed lookup. No write paths touched.

checkout along with this changelog.

## 11. Re-run staleness safety net — STALE badge (piece 1 of 2)

**Context:** re-running an agent makes every agent *transitively
downstream* of it outdated — their stored reports were built on the
**previous** version. Today the system handles only the *error* half:
if a re-run upstream agent errors, the `required_upstream` gate skips
downstream agents cleanly. But when the upstream re-run *succeeds*,
downstream output just sits on its card looking green and fresh while
silently built on the old version — a silent regression.

This is **piece 1 of 2** of the fix: the *safety net* — detect and
**flag** staleness. Piece 2 (an opt-in cascade re-run) is deliberately
separate.

**Change A — `downstream_of()` graph helper
([backend/agents/supervisor.py](backend/agents/supervisor.py)).** A
reverse-graph BFS over `AgentSpec.upstream`: given an agent, returns the
full transitive set of agents that depend on it. Cycle-safe via a
visited guard. This is the exact "blast radius" of a re-run.

**Change B — mark downstream STALE on re-run
([backend/main.py](backend/main.py)).** New `_mark_results_stale()`; the
`/api/run-selected` endpoint now computes
`⋃ downstream_of(a) for a in selected` minus the selected set, and tags
each such agent's stored `_results` entry with `stale: true` +
`stale_reason`. Marking only — **no auto-cascade**. `/api/results`
already returns the `_results` entries verbatim, so the flags reach the
frontend with no endpoint change. The flag is self-clearing: the next
fresh run of that agent overwrites its `_results` entry (via
`_save_partial`), which carries no `stale` key.

**Change C — STALE badge in the Hub ([frontend/app.js](frontend/app.js),
[frontend/style.css](frontend/style.css)).** New `paintStaleChip()` and
a `stale-chip` span in the agent-card header (beside the repair /
coverage chips). A `⚠ stale` amber badge paints — with the upstream
re-run named in its tooltip — whenever a result carries `stale: true`.
Hooked into `renderFinalReport` (run-end render, all agents),
`completeReattachedCard` (page reload / reattach), and cleared in
`completeCard` (the agent finished a fresh run).

**Effect:** re-run migration-planner (or any agent) and its downstream
cards immediately wear a `⚠ stale` badge — the reviewer can no longer
mistake outdated output for current. The data is never auto-discarded
or auto-rebuilt; the user stays in control.

**Not in this piece:** the opt-in cascade ("re-run X **and** everything
below it"), the pre-re-run blast-radius confirmation dialog, and
hash-based invalidation of reviewer approve/reject decisions on changed
code-generation files. Those are piece 2.

**Risk:** low — `downstream_of()` is a pure read-only graph function;
the marking is in-memory only (resets on backend restart, acceptable
for an advisory signal); the badge is render-only. No run/scheduling
behaviour changes.

**Sync:** [backend/agents/supervisor.py](backend/agents/supervisor.py),
[backend/main.py](backend/main.py), [frontend/app.js](frontend/app.js)
and [frontend/style.css](frontend/style.css) copied to the desktop
checkout along with this changelog.

## 12. Re-run cascade dialog + hash-verified review decisions (piece 2 of 2)

Completes the re-run staleness feature begun in §11. Piece 1 was the
*safety net* (detect + flag). Piece 2 is the *control surface*: let the
user act on the blast radius, and stop a stale "approved" verdict from
surviving a code regen.

**Change A — blast-radius endpoint
([backend/main.py](backend/main.py)).** New
`GET /api/agents/{agent_id}/downstream` returns the transitive
downstream set via `downstream_of()` — pure graph, no session state.

**Change B — cascade re-run dialog ([frontend/app.js](frontend/app.js),
[frontend/style.css](frontend/style.css)).** The card `↻ Re-run` button
no longer fires immediately. It first fetches the agent's downstream
set, intersects it with agents that have already produced output, and —
if any — shows `showRerunDialog()`, a 3-choice modal built dynamically
in JS (no index.html change):

- **Re-run all N** — cascade: dispatch this agent + every downstream
  agent to `/api/run-selected`; the backend's `compute_waves` re-layers
  them into dependency order.
- **Just this one** — dispatch only this agent; piece-1 marking flags
  the downstream cards STALE.
- **Cancel** — abort.

No auto-cascade ever happens without the user choosing it. When the
cascade set is dispatched, every stale agent is in the run subset, so
piece-1's `_mark_results_stale` correctly marks nothing.

**Change C — hash-verified review decisions
([backend/review.py](backend/review.py),
[backend/main.py](backend/main.py)).** A reviewer's `approved` verdict
on a converted file must not survive that file being rewritten by a
code-generation re-run. `FileReview` gains a `target_sha` field;
`refresh_from_mappings` preserves it. New
`invalidate_changed_decisions(session_id, current_hash_of)`: for every
file carrying a real verdict it either **backfills** `target_sha` from
the current content hash (establishing a baseline — covers pre-feature
and bulk "approve all" verdicts) or, if the stored hash no longer
matches, **resets the verdict to `pending`** with an explanatory note.
The `/api/review/{sid}/files` endpoint runs this every load, hashing
each row's converted target file(s) via the new `_review_target_hash()`
helper. A verdict on an unchanged file is untouched; a verdict on a
changed file is reset so it is re-reviewed — no stale "✅ approved" on
regenerated code.

**Effect — the full feature (§11 + §12):** re-running an agent now (a)
shows its exact blast radius, (b) offers cascade vs. mark-stale vs.
cancel, (c) paints a `⚠ stale` badge on every downstream card left
unre-run, and (d) auto-invalidates any reviewer verdict whose converted
file changed underneath it. Stale output can no longer masquerade as
fresh, and stale approvals can no longer ship.

**Risk:** low–moderate. The endpoint and `downstream_of()` are
read-only; the dialog is render-only and falls back to re-running just
the agent if the downstream fetch fails. The decision-hash check writes
the review store only when a verdict actually backfills or resets, runs
inside the existing `_lock`, and is wrapped in a non-fatal try/except —
a hash failure degrades to "leave the verdict alone". `target_sha` is
additive (old review JSON hydrates with the `""` default).

**Sync:** [backend/main.py](backend/main.py),
[backend/review.py](backend/review.py), [frontend/app.js](frontend/app.js)
and [frontend/style.css](frontend/style.css) copied to the desktop
checkout along with this changelog.


## 13. Report scrubber missed "all the context" filler; PDF fell back silently

**Symptom.** The 2026-05-15 `code-analysis` export for session
`c9afaa02f85e` opened with a leaked LLM transition line —
`I now have all the context I need. Let me write the full report.` —
sitting above the real report body. The April 24 golden export of the
same session is clean, so this read as a regression in report quality.

**Root cause — filler scrubber gap.** Every completed agent's output
already passes through `scrub_report()` at
[backend/agents/runner.py](backend/agents/runner.py) (`result_text =
scrub_report(final_result_text.strip())`), and the success log confirms
the leaked text survived the scrub. Two pattern gaps in
[backend/agents/report_scrubber.py](backend/agents/report_scrubber.py)
let it through: (1) pattern 1's qualifier branch recognised
`sufficient` / `enough` / `a thorough` / `a complete` … but **not the
bare `all the`** form ("all the context"); (2) pattern 2 only matches
`Let me write …` at the **start of a line**, but here it was a second
sentence on the same line. Result: the whole line was neither a pattern-1
nor a pattern-2 hit and passed straight through to the saved `.md` /
`.docx` / `.pdf`.

**Fix — filler patterns.** Pattern 1's qualifier branch now also accepts
`all` / `all of` / `all the` plus an optional-article adjective form
(`the necessary`, `complete`), and the noun list gained `knowledge`. A
belt-and-suspenders exact-phrase clause was added to pattern 4 for
`I (now) have all (of) (the) context|information|details|signals|knowledge
(I need)`. Verified: the four filler variants are stripped and a real
sentence — `I have all the booking records loaded into the new schema.`
— is left untouched (its noun is not in the transition-noun list).

**Root cause — PDF dropped silently.** `render_agent_pdf()` in
[backend/agents/export.py](backend/agents/export.py) only fell back to
WeasyPrint on a Playwright **package** `ImportError`. When the package is
installed but the **Chromium binary is missing** (`playwright install
chromium` never run), `p.chromium.launch()` throws a different error that
propagated into the auto-export caller's generic `except Exception`
([backend/main.py](backend/main.py), `_auto_export_session`) and the PDF
was silently skipped — which is why the May 15 exports have `.md` +
`.docx` but no `.pdf`.

**Fix — PDF fallback + Chromium.** `render_agent_pdf()` now wraps the
Chromium launch in a try/except; a launch failure falls back to the
WeasyPrint renderer instead of bubbling up and being swallowed. The
Chromium browser binary was also installed into the backend venv
(`playwright install chromium`, build v1134) so the primary
Chromium-fidelity path produces PDFs again.

**Not a bug — single-agent runs.** The May 15 exports contain only
`code-analysis` (vs. the 13-agent April set) because `logs/agents/` shows
those timestamps ran `code-analysis` alone — deliberate single-agent
runs, not an agent-selection regression. `AGENT_REGISTRY` in
[backend/config.py](backend/config.py) still defines all 13 agents and a
full run produces the complete set. No code change here.

**Risk:** low. The scrubber change only widens a delete-on-match regex
for transition filler and was verified not to eat substantive prose; the
PDF change only adds a fallback on a path that previously failed
outright. Both are idempotent and non-fatal.

**Sync:** [backend/agents/report_scrubber.py](backend/agents/report_scrubber.py)
and [backend/agents/export.py](backend/agents/export.py) copied to the
desktop checkout along with this changelog.


## 14. file_map.json "unavailable (HTTP 404)" when the file was on disk

**Symptom.** The Migration Map panel on the migration-planner card showed
`file_map.json unavailable (HTTP 404). The migration-planner agent must
finish before this data is available.` — even for session `c9afaa02f85e`,
whose `file_map.json` (589 mappings) is sitting on disk at
`uploads/c9afaa02f85e/context/file_map.json`.

**Root cause — backend, two gaps in `get_session_file_map`.** The
`/api/sessions/{id}/file-map` endpoint
([backend/main.py](backend/main.py)) resolved the session differently
from its sibling `_load_file_map_for_review`:
(1) it read the in-memory `_session_dirs.get(session_id)` directly —
which is empty after every backend restart until lazily repopulated — so
a restart produced a false `Session not found` 404 instead of
rehydrating from disk; (2) it derived the context dir as
`project_dir.parent / "context"` and checked **only that one path**,
whereas the planner / code-generation pipeline writes the authoritative
file under `source/context/` on some layouts and `<root>/context/` on
others. `_load_file_map_for_review` already handles both correctly by
using the canonical `_session_folder()` and a two-candidate loop.

**Fix — backend.** `get_session_file_map` now calls `_require_session()`
(rehydrates a disk-resident session instead of 404'ing) and resolves the
file via `_session_folder()` with the same `source/context/` →
`context/` candidate loop as `_load_file_map_for_review`. Verified
against `c9afaa02f85e`: `context/file_map.json` resolves and parses
(589 mappings).

**Root cause — frontend.** In [frontend/app.js](frontend/app.js) the
file-map panel's `toggle` handler set `loaded = true` *before* the fetch.
A failed load therefore latched permanently — collapsing and reopening
the panel would not retry, so even after migration-planner finished the
panel stayed stuck on the error until a full page reload. The error text
was also hard-coded to blame the planner regardless of the real HTTP
status.

**Fix — frontend.** `loaded` is now set to `true` only after a
successful render, so a collapse + reopen retries once the agent
finishes. The error box surfaces the backend's actual `detail` string
when present, falling back to the planner message only when no detail is
available.

**Not a bug — session `3ded799af84f`.** That session has no `context/`
directory at all, so a 404 there is legitimate — migration-planner has
not run for it. The retry fix means its panel will populate on reopen
once the agent completes.

**Risk:** low. Backend change aligns one endpoint's path resolution with
an existing, proven helper; frontend change only moves a flag assignment
and widens an error message. No schema or contract change.

**Sync:** [backend/main.py](backend/main.py) and
[frontend/app.js](frontend/app.js) copied to the desktop checkout along
with this changelog. Restart the desktop backend and hard-reload the
frontend for the fix to take effect.
