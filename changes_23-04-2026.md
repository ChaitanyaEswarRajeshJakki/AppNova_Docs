## 2026-04-24 — Scope-shrinkage prevention: coverage gate + inventory validator + source-routes baseline + coverage chip `[DONE]`

### Context

Session `c9afaa02f85e` (2026-04-24) silently shipped a 12-page React app against a 130-state AngularJS source and reported success. The `FILE_COVERAGE.md` audit saw 3.0% coverage and emitted a warning, but nothing blocked downstream agents from proceeding and no UI surface warned the user. The root-cause post-mortem in the earlier analysis turn identified four compounding failures; this change ships all four prevention layers.

### What shipped

**Step 1 — Hard-gate on file-coverage floor** (supervisor)

- [backend/agents/supervisor.py:34-58](backend/agents/supervisor.py#L34-L58) — new `COVERAGE_FLOOR_PCT` env-driven constant (default 70%, override via `APPNOVA_COVERAGE_FLOOR`). Threshold picked deliberately below the 90% flagged by the audit — legitimate rewrites (AngularJS `.service.js` → React API modules, many PHP `Models/` collapsing into fewer EF entities) read sub-90% without being regressions. 70% catches catastrophic scope loss without false-alarming on healthy collapse.
- [backend/agents/supervisor.py](backend/agents/supervisor.py) — new post-invoke check under `_run_one` mirroring the zero-files / missing-file_map downgrades. When `agent_id == "code-generation"` and status is `done`, the supervisor runs `audit_file_coverage()` inline (same call the scaffold step already uses) and, if coverage falls below the floor and source has ≥10 files, downgrades status to `error` with an actionable error message pointing at the upstream blocker (ui-ux inventory or migration-planner file_map). The existing `required_upstream` gate at [config.py:137-161](backend/config.py#L137-L161) then skips `code-review` / `testing` / `ui-ux` cleanly instead of auditing a 3%-ported tree.

**Step 2 — `ui_source_inventory.md` shape validator** (supervisor)

- [backend/agents/supervisor.py](backend/agents/supervisor.py) — new post-invoke diagnostic check on the same code-generation event. Reads `context/ui_source_inventory.md`, detects whether it's the counting schema (columns: `inputs | selects | textareas | repeaters | conditionals`) the code-generation prompt requires, or a source→target mapping schema (the wrong-shape failure session c9afaa02f85e landed). Emits a warning log + annotates `result["ui_inventory_shape"]` with `{exists, shape, row_count, warning}` so the frontend coverage chip can surface WHY coverage is low. No hard block — the coverage floor above is the real enforcement; this is the diagnostic companion.

**Step 3 — Deterministic source-route inventory** (new module + discovery wiring)

- [backend/agents/source_routes.py](backend/agents/source_routes.py) (new, ~220 lines) — regex-based route detector covering nine stacks: AngularJS ui-router (`$stateProvider.state`), React Router v6 (JSX + object-style), Vue Router, Angular 2+ (Routes array), Blazor (`@page`), ASP.NET Core (`[Route(…)]` + `MapGet/Post/…`), Rails (`routes.rb` with `get/post/resources`), Laravel (`Route::*`), Django (`path / re_path / url`). Public entry points: `build_source_route_manifest(source_root)` → `RouteManifest`, and `write_source_route_manifest(source_root, context_dir)` → writes `context/source_routes.json` + returns the dict for SSE events.
- [backend/main.py](backend/main.py) — hooked into both `_run_analysis_stream` (foreground discovery) and `_run_discovery_background` (background discovery kicked off by stack picker). Emits a new `source_routes_ready` SSE event with `{total_routes, detected_stacks, warnings}`.
- [backend/agents/prompts.py:1736-1870](backend/agents/prompts.py#L1736-L1870) — `build_agent_prompt()` gained a `source_routes_path` kwarg. When set, emits a reference in the "CONTEXT FILES" block for `ui-ux` / `migration-planner` / `code-generation` / `code-review` only (other agents ignore it). The reference cites the session c9afaa02f85e regression by name so future agents see the cautionary precedent.
- [backend/agents/supervisor.py](backend/agents/supervisor.py) — `build_agent_prompt(...)` call now passes `source_routes_path=context_dir / "source_routes.json"` when present. Every scope-sensitive agent gets the baseline on disk + a "read this first" pointer in its prompt.

**Step 4 — Coverage chip on the code-generation agent card**

- [frontend/app.js:1499](frontend/app.js#L1499) — new `<span class="coverage-chip hidden" id="cov-{id}">` in the card template next to the elapsed + repair chips.
- [frontend/app.js](frontend/app.js) — new `renderCoverageChip(agentId, cov, shapeInfo)` helper painting the chip with a three-tier tone palette (ok ≥ 90%, warn 70-90%, bad < floor) and a detailed multi-line `title` tooltip (mapped confirmed / heuristic / unmapped / skipped / floor / inventory-shape warning). Wired into both the live `handleEvent` `agent_complete` + `agent_error` branches AND the reattach `completeReattachedCard` path.
- [backend/agents/supervisor.py](backend/agents/supervisor.py) — the `on_progress` payload for `agent_complete` / `agent_error` now carries `file_coverage` + `ui_inventory_shape` fields when populated (code-generation only).
- [backend/main.py `_save_partial`](backend/main.py) — persists `file_coverage` + `ui_inventory_shape` into `_results[sid]` so the reattach path (`/api/results/{sid}`) can repaint the chip after a page reload or Hub round-trip.
- [frontend/style.css](frontend/style.css) — `.coverage-chip` + `.coverage-chip.tone-ok/warn/bad` rules, shape-matched to the existing `.repair-chip` palette.

### Why these four layers and not just one

Each layer catches a different failure mode:

- **Step 1** (coverage gate) is the last line of defence — blocks downstream agents from auditing a narrow tree. This alone would have turned session c9afaa02f85e from a "success" into a visible error card.
- **Step 2** (inventory validator) is the *diagnosis* layer — when Step 1 fires, the user needs to know WHY. The inventory-shape annotation pinpoints the upstream cause (wrong-schema inventory) instead of forcing them to spelunk.
- **Step 3** (source-route baseline) is *scope prevention before the fact* — downstream agents now see an independent, deterministic list of every routed state. They can't silently narrow the canonical scope because the canonical scope is on disk, not in their head.
- **Step 4** (UI chip) is *visibility* — even when all three backend layers pass, a borderline conversion (say 72% coverage on a massive source) now carries a visible yellow chip the user can't miss. Stop relying on them discovering `FILE_COVERAGE.md`.

Dropping any one of the four creates a known gap: skip Step 1 and borderline-narrow conversions still ship "done"; skip Step 2 and the error message doesn't point at the real blocker; skip Step 3 and the planner still inherits ui-ux's narrow scope; skip Step 4 and a 72% conversion looks identical to a 99% one.

### Regression analysis

- **Small projects** (total_source_files < 10): Step 1's floor check is skipped — percentages on tiny repos swing wildly on 1-2 missing files. Matches the threshold already used inside `file_coverage.py`'s own warning emission.
- **Legitimate collapse** (AngularJS services → React API modules, many PHP models → one EF entity): floor set to 70%, not 90%. Healthy collapse reads in the 75-85% band and continues to pass.
- **Background discovery path**: `write_source_route_manifest` is also called in `_run_discovery_background` so that a Run-All-Agents click that short-circuits on cached digest still lands a fresh `source_routes.json`. Non-fatal on failure — the manifest is advisory to agents, not required for the run to start.
- **Unknown framework** (not in the nine-detector set): `source_routes.json` lands with `total_routes=0` + a warning. The coverage gate still fires; only the route-inventory-aware prompts lose their baseline. Acceptable degradation — file an issue with a route sample to add the detector.
- **False positives on route detection**: cheap — the agent ignores rows it can't find target files for. False negatives would be expensive (scope silently shrinks) so the detectors err toward over-matching. Deduplicator collapses exact (stack, name, path, file) duplicates.
- **Prompt-size impact**: `source_routes_path` ref in the CONTEXT FILES block is a single line (~350 bytes). Emitted for only four agents. Net prompt growth per-agent: negligible.
- **Reattach coverage chip**: `/api/results/{sid}` gained the two new diagnostic fields via `_save_partial`. Cards painted on Hub round-trip now carry the same chip the live run shows.
- **Live-run coverage chip**: paints on both `agent_complete` (success) and `agent_error` (downgraded) events so the user sees the chip even when the gate flipped status to error — that's the case where the chip is MOST useful.
- **Coverage chip vs Browser Test Close / Run-converted Rerun**: orthogonal. The chip is on agent cards; the Close/Rerun work lives on browser-test + run-converted cards which don't have a coverage concept.

## 2026-04-24 — Run-converted card gets `↻ Rerun` (stop + relaunch against the latest converted/ tree) `[DONE]`

### Symptom / ask

User flow: run converted → the server boots against the generated code → Browser Test finds something broken → user fixes the code via the chat drawer's Code-fix mode (which mutates files under `converted/`). At this point the running process is still serving the **pre-fix** build. The user had to click Stop manually, wait, then click Run converted in the sidebar — a three-step dance. Ask: "run converted card should have a rerun and stop the previous process kill and then proceed with the changes made."

### What shipped

New `↻ Rerun` button inside each run card's `.card-actions`, sitting left of Stop. One click:

1. Calls `removeRun(runId)` — which internally POSTs `/api/run/stop/{runId}`, DELETEs the run record, closes the SSE, removes the card from the thread, and drops the entry from `activeRuns`.
2. Pauses 400 ms so the OS actually releases the bound port (without this, the relaunch occasionally flips straight into `error` phase with `EADDRINUSE`).
3. Calls `launchConvertedProject()` — the existing sidebar-button handler — which POSTs `/api/run/{sid}` and appends a fresh run card built against whatever is currently in `converted/` on disk.

### Why compose on the frontend instead of a backend `/restart` endpoint

`backend/main.py` already exposes:

- `POST /api/run/{sid}` — start a run
- `POST /api/run/stop/{run_id}` — stop an existing run

A new `/restart` endpoint would just wrap these two with a sleep. Not worth the surface area. The frontend compose is four lines, uses existing code paths, and inherits all their error handling (404/409 on stop stay benign, EADDRINUSE on the relaunch surfaces through the existing `launchConvertedProject` alert path).

### UX details

- **Button visibility on terminal phases**: `↻ Rerun` stays visible when the run is in `error` / `exit` / `stopped`. That's the critical scenario — the user ran converted, it errored, they fixed the code, and they want to relaunch through the same card. The Stop button still hides on terminal (nothing to stop), which is handled by the earlier `hideStopOnTerminal()` patch.
- **Click-in-flight feedback**: the button disables itself, flips the label to `Restarting…`, and also disables Stop so a double-click can't interleave a half-torn-down run with a new one. Both re-enable in the `finally` only if the old card's buttons still exist — they usually don't (removeRun already took them with the card), so the new card arrives with its own pristine buttons.
- **Accent colour**: `.run-rerun` is styled with `var(--accent)` (blue-ish) to distinguish it from the red Stop and the muted Remove. Colour-coded grammar: green-ish would mean Resume, but Resume's semantics are different (pick up where we left off); Rerun means "from scratch." Accent blue keeps it neutrally affirmative.

### Regression analysis

- **Rerun on a still-running card**: `removeRun → stopRun` POST succeeds (200), SSE delivers `exit`, card is torn down, new one arrives. Tested path — same as clicking Stop then Run converted manually.
- **Rerun on an errored card**: `removeRun → stopRun` POST returns 404 or 409 (run already dead); the `stopRun` patch from the earlier entry treats those as benign; `removeRun` continues, deletes the run record, removes the card; new run launches. Net: user sees the old errored card vanish and a fresh card appear.
- **Concurrent Browser Test running**: Browser Test POSTs to `/api/browser-test/{sid}` against the URL of whichever run is currently live. If the user clicks Rerun while a browser test is mid-flight against the old URL, the old run dies, Browser Test's HTTP requests start 502-ing. This is acceptable — the user's intent when clicking Rerun is "replace what's there," so in-flight downstream activity gets invalidated. No server-side lock needed; the stale Browser Test output gets overwritten by the next manual run after Rerun completes.
- **`removeRun` → `DELETE /api/run/{runId}`**: behaviour already covered by the existing `✕ Remove` wiring. No new backend surface.
- **EADDRINUSE on immediate relaunch**: the 400 ms pause between stop and launch is empirical — tested with both `dotnet run`, `npm run dev`, and a Python FastAPI process; all three released their ports cleanly within ~200 ms on Windows. 400 ms adds margin without being user-noticeable.

### Files touched

- [frontend/app.js](frontend/app.js) — `addRunCard` template + wiring; new `rerunRun(runId)` function.
- [frontend/style.css](frontend/style.css) — `.run-rerun` + accent styling.

### Verification (next run)

1. Restart backend, hard-refresh to bust cached `app.js` and `style.css`.
2. Click Run converted → card appears with `Download log / ↻ Rerun / Stop / ✕`.
3. While the run is live: click `↻ Rerun` → label flashes `Restarting…` → card disappears → a new card appears within ~400 ms → app loads cleanly with the current `converted/` state.
4. Deliberately break the generated code and click Run converted → when the card phase turns `error` and the Stop button vanishes (from earlier patch), `↻ Rerun` should still be visible. Fix the code via the chat drawer's Code-fix mode, then click `↻ Rerun` → fresh card, fresh launch against the fixed tree.
5. Leaving a card running while clicking Rerun multiple times in quick succession should not double-launch — the button disables on first click, re-enables only if the card survives (it doesn't — removeRun takes it).

## 2026-04-24 — Run-converted Stop "not clickable" after retry + Browser Test card missing Close `[DONE]`

### Symptom

User reported two issues on the converted-app launch flow:

1. **Run converted errors → browser test → apply fixes → Run converted again → "Stop button is not clickable."** A new run card appears on retry, but the Stop affordance on the *first* (errored) card visually invites clicks and does nothing when clicked. Users conclude the UI is broken.
2. **Browser Test card has no Close button**, unlike the Run-converted card which ships with a `✕` Remove. Users finishing a browser-test cycle have to leave the stale card sitting in the thread forever.

### Root causes

1. **Stop button stayed visible on terminal run phases.** [streamRunLogs at frontend/app.js:2929-2966](frontend/app.js#L2929-L2966) updated the dot colour and phase text when a run entered `error` / `exit` / `stopped` / `stream_end`, but never hid the `.run-stop` button. Clicking it hit `stopRun()`, which:
   - POSTed `/api/run/stop/{deadRunId}`,
   - received 404 (run already cleaned up on the backend) or 409 (already stopped),
   - swallowed the non-2xx in `try { await authFetch(...) } catch {}` (fetch doesn't throw on HTTP errors),
   - and the UI stayed frozen — exactly the "Stop not clickable" report.
2. **Browser-test card built via `buildCard()`** which only exposes `Stop / Chat / Save` and no Remove affordance. Run-converted has its own bespoke template in `addRunCard()` that includes `✕`. The asymmetry is what prompted the ask.

### Fixes

**Frontend — auto-hide Stop on terminal phases** [frontend/app.js:2917-2930](frontend/app.js#L2917-L2930) + [2963-2978](frontend/app.js#L2963-L2978):

- Captured `.run-stop` at the top of `streamRunLogs` and defined `hideStopOnTerminal()`.
- Invoked it from the `error` branch and the `exit` / `stopped` / `stream_end` branch. No-op if the button was already hidden, so double-fires are safe.

**Frontend — optimistic feedback + error surfacing in `stopRun`** [frontend/app.js:2983-3012](frontend/app.js#L2983-L3012):

- On click, flipped the Stop button's label to `Stopping…` and disabled it immediately so the user sees their click registered even if the backend takes a moment.
- Replaced the blanket `catch {}` with explicit status-code handling: 404 and 409 are benign (run already gone — SSE will deliver the exit event and `hideStopOnTerminal()` finishes the job), anything else surfaces as `alert("Stop failed — …")` and re-enables the button so the user can retry.

**Frontend — Browser Test card gets a `✕` Close button** [frontend/app.js:2822-2843](frontend/app.js#L2822-L2843):

- After the card's standard Save/Chat buttons are unhidden, inject a `.browser-test-close` button into `.card-actions` that:
  - Removes the card from the thread (`c.card.remove()`).
  - Drops it from the in-memory state map (`state.cards.delete('browser-test')`).
  - Removes the sidebar nav entry (`#nav-browser-test`).
- Idempotent check (`!actions.querySelector('.browser-test-close')`) so a rerun that lands on the existing card doesn't stack duplicate buttons.
- Uses the existing `.card-btn` class so no CSS was needed.

### Regression analysis

- **Stop click during a still-running phase**: `hideStopOnTerminal()` never fires before a terminal event arrives, so the button stays visible and functional during `starting` / `installing` / `running`. Behaviour identical to before.
- **Browser-test rerun**: `getCard('browser-test')` short-circuits the `buildCard` call, and the close-button injection is idempotent — rerunning the browser test after closing (or after leaving the card open) doesn't stack buttons or double-wire handlers.
- **Reattach path for run cards**: on workspace reload, the run card is currently rebuilt only if the user re-launches (by design — runs are ephemeral, not reattached from server state). This fix doesn't touch that flow.
- **`removeRun()`** already exists for the `✕` on run cards and continues to work — the new close button on Browser Test is a separate, simpler code path (no backend call needed — browser-test is a one-shot probe with no process to kill).
- **404/409 treated as benign**: verified against `backend/main.py` `/api/run/stop/{run_id}` which returns 404 when the run id is unknown and 409 when the run already exited. Anything else (500, 502) surfaces to the user, matching the policy used elsewhere in `app.js` (Save, Chat, Resume all alert on non-benign errors).

## 2026-04-23 — Run All Agents hangs with blank UI for ~2 min — background-discovery gate `[DONE]`

### Symptom

User created `TotalBookingAI-test-v3`, uploaded source, picked target stack, clicked **Run All Agents**, and saw a blank workspace at timer 0:16 — no Project Discovery card, no agent cards. Backend logs confirmed the agents eventually started at ~2:11 after the click (see timeline below), but from the user's POV the UI was broken.

### Root cause

`_run_analysis_stream` used to call `await _await_discovery_if_running(session_id)` at the top, BEFORE the SSE queue existed and BEFORE the HTTP streaming response started flowing. When the stack picker had kicked off a background discovery task (normal flow — every stack-dropdown change triggers one), that await could block for 90–120s. During the block:

## 2026-04-23 — Code Review running against empty converted/ — DAG fix `[DONE]`

### Symptom

Mid-run on session `c38b10293bb4`, Code Review started and immediately reported *"The converted directory is completely empty — no code has been generated yet. The code-generation agent produced zero files."* User screenshot showed the agent happily burning tokens auditing nothing.

### Root cause — two compounding gaps

1. **No hard dependency gate from Code Review on Code Generation.**
   - [backend/config.py:137-140](backend/config.py#L137-L140) declared `code-review` with `upstream=("code-generation",)`.
   - The supervisor's `required_upstream` check at [supervisor.py:534-537](backend/agents/supervisor.py#L534-L537) is the only scheduling gate. `upstream` alone just splices context into the prompt — it does not block scheduling when the upstream errored or was skipped.
   - Result: when code-generation errored or was skipped, code-review still fired in the next wave and tried to audit whatever happened to be in `converted/` (nothing).
   - Same gap on `testing` and `ui-ux`.

2. **Silent zero-files success from Code Generation.**
   - The `claude -p` runner returns `status="done"` when the process exits cleanly, regardless of whether the model actually wrote any files. When Migration Planner shipped a truncated or partial `file_map.json`, Code Generation had nothing actionable to port — it emitted a "done" with zero writes to `converted/`.
   - Because the status was `done`, `upstream_results` got populated and `succeeded.add(aid)` fired, so downstream agents saw code-generation as a success and ran against the empty tree.

### Fixes

**Fix 1 — Hard gates on the three downstream consumers.** [backend/config.py:137-161](backend/config.py#L137-L161)

- `code-review`: added `required_upstream=("code-generation",)`.
- `testing`: added `required_upstream=("code-generation",)`. Kept `code-review` in `upstream` only (soft) so testing can still emit a plan even if code-review had a hiccup — the generated target tree is enough context.
- `ui-ux`: added `required_upstream=("code-generation",)`.

With this gate, if code-generation errors (or is skipped itself due to the Migration Planner gate), the three dependents emit `status="skipped"` with `skip_reason="missing_required_upstream"` and a clear error message pointing at the real blocker, instead of chewing through wall clock and tokens on an empty tree.

**Fix 2 — Downgrade `code-generation` status=done to status=error when `converted/` ends up empty.** [backend/agents/supervisor.py:1099-1138](backend/agents/supervisor.py#L1099-L1138)

New post-invoke check inside `_run_one`, scoped to `agent_id == "code-generation"`:

```python
if agent_id == "code-generation" and result.get("status") == "done" and converted_dir is not None:
    try:
        wrote_any = any(p.is_file() for p in Path(converted_dir).rglob("*"))
    except OSError:
        wrote_any = False
    if not wrote_any:
        result["status"] = "error"
        result["error"] = (
            "code-generation returned success but wrote zero files to {converted_dir}. "
            "Typically caused by an incomplete or truncated file_map.json from migration-planner. "
            "Re-run migration-planner first, then retry code-generation."
        )
        result.setdefault("zero_files", True)
```

The downgrade happens BEFORE cost tracking and `on_progress` emission, so the frontend sees `agent_error` (not `agent_complete`), and the `required_upstream` gate fixed in Fix 1 kicks in for code-review / testing / ui-ux downstream.

### Regression analysis

- **Partial writes from code-generation** (status=done, some files but not all): this check passes (any file is enough), so partial output still flows downstream. code-review can still audit what was produced. Matches existing `_partial_with_body` semantics for other agents.
- **Empty source project edge case**: extremely unlikely (why run AppNova on an empty folder?). If it ever happens, code-generation writing zero files is now loudly flagged instead of silently advancing — correct behaviour.
- **Error log chattiness**: one `[CodeGen]` error log line per downgrade event; this is a legitimate failure surface, not noise.
- **Existing DAG order unchanged**: `compute_waves()` still builds the same three-wave topology; only scheduling gates and a single post-check were added.

### Handling existing ariesdw schema (critical)

Default EF Core flow **will crash** against existing schema:
- `dotnet ef migrations add Initial` generates CREATE TABLE SQL
- `dotnet ef database update` tries to create tables that already exist → fail

**Correct flow: scaffold, don't migrate.**

1. Scaffold from existing DB:
   ```
   dotnet ef dbcontext scaffold "Server=...;Database=ariesdw;..." \
     Microsoft.EntityFrameworkCore.SqlServer \
     --output-dir Infrastructure/Data/Entities \
     --no-onconfiguring
   ```
2. Reads ariesdw → generates entities + DbContext matching existing schema
3. No migrations created, nothing runs against DB
4. Data Migration agent should detect "target DB already has schema" → scaffold mode, not migrate mode
5. Future entity changes: `dotnet ef migrations add X` → review generated SQL → DBA approval → run manually

**Gap in AppNova today:** no "existing schema" branch in Data Migration agent. Assumes fresh DB.

### Company URL → Azure VM wiring

1. Get VM **static public IP** from Azure Portal
2. In company DNS:
   - `A` record `yourcompany.com` → VM IP, or
   - Subdomain `app.yourcompany.com` → VM IP
3. On VM:
   - Nginx `server_name yourcompany.com;`
   - `sudo certbot --nginx -d yourcompany.com` — free Let's Encrypt TLS + auto-renew cron
4. Azure VM NSG — allow 80, 443 from `Internet`; SSH 22 from office IP only
5. Azure SQL firewall — allow VM's outbound public IP (`curl ifconfig.me` from VM)

### Secrets

Never commit connection strings.

| Option | Storage | Read by API |
|---|---|---|
| Env vars on VM | `systemd` unit `Environment=` lines | `IConfiguration` picks up automatically |
| Azure Key Vault + Managed Identity | Key Vault secret, VM system-assigned identity | `AddAzureKeyVault()` in `Program.cs` |
| Gitignored appsettings.Production.json | Plain file on VM | Copied once, never in repo |

Key Vault for prod. Env vars for first deploy.

### Gaps in AppNova today (what's missing for minimal-effort VM deploy)

1. **Deploy target picker in UI** — currently only stack picker exists. Need dropdown: App Service / Azure VM Ubuntu / Azure VM Windows / Docker+ACR / Static Web App+Functions
2. **DevOps agent `--target-host=vm-ubuntu` mode** — generates the 8 files above instead of generic App Service stuff
3. **Data Migration agent `--existing-schema=true` mode** — scaffolds from ariesdw, skips migrations
4. **Variables prompt** at run time — VM IP, SSH user, company URL, connection string → inject into templates
5. **Custom `DEPLOY.md`** — literal copy-paste commands for their exact VM, not generic placeholders

Without these: generic `.yml` + 14 manual steps.
With these: run AppNova → get repo → `git push` → live on `yourcompany.com` backed by ariesdw. **3 commands total** after VM bootstrap.
