## fix(chat): raise the chat-turn timeout ceiling to a hard maximum of 4 hours

**What**
- **Run-converted dev-chat** ([backend/agents/dev_assist.py](backend/agents/dev_assist.py#L73-L82)): the per-turn Claude Code subprocess timeout was `DEV_ASSIST_TIMEOUT` defaulting to **300s (5 min)**. Introduced `_DEV_ASSIST_TIMEOUT_CAP_SECONDS = 4*60*60` and made the effective value `min(env_or_default, cap)`, so the default is now **4 hours** and any `DEV_ASSIST_TIMEOUT` override is clamped so it can never exceed 4 h. The caller-passed `timeout_seconds` in `run_turn` is clamped the same way ([dev_assist.py:409](backend/agents/dev_assist.py#L409)).
- **Review/recovery chat** ([backend/agents/chat.py](backend/agents/chat.py#L54-L61)): `chat_turn(...)`'s `timeout` default was **600s (10 min)**. Added module constant `_CHAT_TURN_MAX_TIMEOUT = 4*60*60`, used it as the default, and clamp `timeout = min(int(timeout), _CHAT_TURN_MAX_TIMEOUT)` inside the function. This governs both the interactive report/code chat and the background coverage/parity recovery turns (`_schedule_recovery_chat_turn`).
- **Error copy** ([backend/main.py:6747](backend/main.py#L6747)): the dev-chat `504` handler hard-coded *"exceeded the 5-minute LLM timeout"*. Replaced with a message that reports the actual configured ceiling (`{h}h{mm}m / {s}s`) so it never drifts from the real value again.

**Why**
- Code-mode chat turns can legitimately run long ‚Äî a single turn may `npm install` / `dotnet restore`, build, and iterate on a fix before responding. At 5 min (dev-chat) and 10 min (review chat) those turns were being killed mid-flight, surfacing as spurious `turn_error` / `504` failures during run-converted iteration. Raising the ceiling to 4 h lets a turn finish; the clamp keeps a runaway/hung turn from living forever.
- Implemented as a **maximum** (clamped), per the request: env/callers may set a *lower* bound but 4 h is an absolute cap. Frontend has no time-based abort (Stop button is user-initiated `AbortController` only), and the SSE stream keeps the socket warm with a 15 s keepalive, so the app-level asyncio timeout is the sole binding constraint ‚Äî no proxy/uvicorn timeout sits in front of the dev loop.

**Files**
- `backend/agents/dev_assist.py`
- `backend/agents/chat.py`
- `backend/main.py`

**Verification**
- `python -m py_compile backend/agents/dev_assist.py backend/agents/chat.py backend/main.py` ‚Üí clean.
