# 2026-06-11 15:05 — Fix Run-Converted `process_crashed` on slow .NET cold build

## Symptom
Browser test: "The Run-converted process for http://127.0.0.1:5050 did not reach
a ready state before the browser test started. Reason: process_crashed."

## Root cause (from the persisted run log `backend/logs/runs/58dd9749df6d/20260611-143456-*.log`)
The converted app is **healthy** — both servers came up:
- backend (`Aries.API`) listened on :5051,
- Vite frontend was "ready in 434ms" on :5050.

But the **generated `run.bat` serialized startup**: it launched the backend
window, then **blocked up to 60s waiting for the backend to bind** before
launching the frontend window. A cold `dotnet run` build + EF seed takes
~30-45s, so port **5050 (the frontend, which the browser test targets) didn't
bind until ~35s** — past the run-manager's `CANDIDATE_TCP_TIMEOUT=45s` window.
The run-manager then killed `run.bat` (rc=1), and its detached-success path only
accepts **rc==0 with the BACKEND reachable first**, so a healthy-but-slow app was
mislabeled `process_crashed`. (The detached child windows survived the kill —
that's why the ports answered afterward.)

Measured after the fix: frontend :5050 bound at **+4s**, backend :5051 at **+34s**
— confirming the frontend is fast and the .NET cold build is the long pole.

## Fixes
### Fix A — launch the frontend in PARALLEL (no blocking backend wait)
`backend/agents/tools/scaffold.py` `_write_split_run_scripts`:
- **run.bat**: removed the `:wait/:ready/:timeout/:fe` netstat loop. The backend
  window AND the frontend window are now spawned back-to-back; run.bat exits
  rc=0 in ~1s. Vite binds :PORT in ~1s and its `/api` proxy retries the backend
  until it's up.
- **run.sh**: removed the `for i in $(seq 1 60)` backend-wait; the frontend
  (foreground `npm run dev`) now starts immediately while the backgrounded
  `dotnet run` warms up.
- Updated the function docstring (was still describing the old `start /B` +
  poll-then-foreground-npm shape).

### Fix B — run-manager accepts a ready FRONTEND as success
`backend/agents/tools/run_manager.py` detached-launcher classification
(`_launch_and_await_ready`, multistack branch): the browser test targets the
frontend, so probe the **frontend first** (20s) and treat `http_ok = fe_ok OR
be_ok` as `running_detached` — instead of requiring the (slow) backend to answer
before the frontend is even checked. Backend probe (HTTP 8s + TCP 4s fallback)
is retained so a backend-only readiness signal still classifies as detached.

### Applied to the live June conversion
Regenerated `uploads/58dd9749df6d/converted/{run.bat,run.sh,_run_backend.cmd,_run_frontend.cmd}`
via the fixed generator (correct CRLF), so the next "Run Converted" uses the
parallel-launch scripts.

## Validation
- `py_compile` clean on both touched files.
- End-to-end relaunch via the new `run.bat` (as `cmd /c run.bat`, like the
  run-manager): **frontend :5050 bound at +4s (HTTP 200)**, backend :5051 at +34s
  (HTTP 404 = reachable, non-5xx). Test processes cleaned up afterward.
- No-regression: backend still launches in its own visible window and still
  tee's to backend.log; `run.sh` still backgrounds dotnet with the EXIT trap.
