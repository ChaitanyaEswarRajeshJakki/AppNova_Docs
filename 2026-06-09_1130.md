# 2026-06-09 11:30 — Fix ASGI receive errors in HTTP capture middleware

## Problem
The backend logged recurring unhandled ASGI exceptions tied to
`_http_capture_middleware`'s request-body replay:

1. `TypeError: object async_generator can't be used in 'await' expression`
   on streaming/SSE endpoints (e.g. `GET /api/session/{sid}/events`).
2. After a first patch attempt: `RuntimeError: Unexpected message received:
   http.request` on streaming POSTs (e.g. `POST /api/run-selected/{sid}`).

Both returned 200 to the client but flooded the terminal/log on every
streaming-response disconnect cycle.

## Root cause
`_http_capture_middleware` drained the request body with
`await request.body()` and then **manually overrode `request._receive`** with
a replay shim so downstream handlers could re-read the body.

That override was both unnecessary and harmful. Starlette's
`BaseHTTPMiddleware` runs the dispatch on a `_CachedRequest`, whose
`wrapped_receive` **already caches the drained body and replays it to the
downstream app automatically** (see `_CachedRequest` docstring in
`starlette/middleware/base.py`). The hand-rolled `_receive` shim fought that
state machine:
- As an async generator (`yield`) → `await request._receive()` raised the
  `async_generator` TypeError.
- As a coroutine returning `http.request` → it desynced the
  `_wrapped_rcv_consumed` disconnect-poll state, so `wrapped_receive` saw an
  `http.request` where only `http.disconnect` was valid → RuntimeError.

## Fix
`backend/main.py` — removed the `request._receive` override entirely. Keep
the `await request.body()` call (needed to capture the body for error logs);
Starlette's `_CachedRequest` handles downstream replay. Added a comment
warning against reintroducing the shim.

## Scope / risk
- Single-function change; no API, schema, prompt, env-var, or agent changes.
- README not updated (no user-facing behaviour change).
- Downstream handlers still receive the request body (verified against
  Starlette's `_CachedRequest.wrapped_receive`, lines 60-67).
