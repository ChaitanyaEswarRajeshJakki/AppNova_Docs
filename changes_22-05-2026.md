# Changes — 22-05-2026

## 1. "Run Converted" failed with "Could not detect target stack"

**Symptom.** Clicking **Run Converted** after approving a review (session
`378d02ddd74c`) popped `Could not start: Could not detect target stack.
Ensure the converted project has a package.json / requirements.txt /
*.csproj / pom.xml / index.html.`

**Root cause — three layers.** A disk audit of `converted/` showed the
project had **no build manifests at all**: no `.csproj` / `.sln` for the
ASP.NET backend (`backend/src/ARIES.{API,Application,Domain,Infrastructure}/`,
~146 `.cs` files), and no `package.json` for the React frontend
(`frontend/src/*.tsx`, ~204 files). Code-generation emitted source files
but never the manifests that make them buildable.
1. **`detect_stack` was shallow** ([backend/agents/scaffold.py](backend/agents/scaffold.py))
   — it inspected only the `converted/` root plus one level, so it could
   not see `backend/src/ARIES.API/Program.cs` or
   `frontend/public/index.html` and returned `"unknown"`.
2. **No manifests** → even a deeper detector has nothing to build.
3. **The post-codegen scaffold step never ran** for this session — no
   `run.sh` / `run.bat` / `tests/` / `migrations/` either.

**Fix — build-manifest generation.** New `ensure_build_manifests()` in
[backend/agents/scaffold.py](backend/agents/scaffold.py) backfills the
missing manifests, idempotently:
- **.NET** — one `.csproj` per project discovered under `backend/src/`,
  with the SDK chosen per project (`Microsoft.NET.Sdk.Web` when the folder
  has `Program.cs` / `Controllers/` / `appsettings*.json`, else
  `Microsoft.NET.Sdk`), `net8.0`, `ProjectReference`s inferred from
  cross-project `using` directives, and a curated, version-pinned NuGet set
  inferred from the `using`s actually present (EF Core + SqlServer +
  Design, JwtBearer, etc.). Plus a `.sln` tying them together. This is
  **best-effort** — it cannot guarantee a clean `dotnet build` (that
  depends on the generated code), but always yields a structurally valid,
  detectable, launchable project.
- **React** — `package.json` (deps scanned from the actual `import`s),
  `vite.config.ts` (dev server honours `PORT` / `BACKEND_PORT`, proxies
  `/api` to the backend), `tsconfig.json`, and a Vite-root `index.html`.
- **Run scripts** — for a `backend/`+`frontend/` split it writes explicit,
  layout-correct `run.sh` / `run.bat` (the generic dual-stack runner
  assumes the `.csproj` sits directly in `backend/`, which is wrong for the
  `src/ARIES.API/` nesting).

**Fix — detection + self-heal.** `detect_stack` now also recurses (via the
existing shallow rglob) for nested `package.json` / `Program.cs` /
`index.html`. `ensure_mandatory_structure` calls `ensure_build_manifests`
before stack detection. `run_manager.start_run`
([backend/agents/run_manager.py](backend/agents/run_manager.py)) now calls
`ensure_mandatory_structure` as a **self-heal** before detecting the
stack — so a converted project whose post-codegen scaffold never ran is
repaired on the next **Run Converted**, without a full re-run of
code-generation.

**Verified** against `378d02ddd74c` (on a temp copy): `detect_stack` →
`dotnet`; generated 4 `.csproj` + `Backend.sln` + `frontend/package.json`
+ `vite.config.ts` + `tsconfig.json` + `index.html` + `run.sh` + `run.bat`,
all structurally correct.

## 2. Review page — full legacy-source tree view

**Symptom.** The Review page's file tree showed only **9** files under its
`source/` node — the user expected to browse the whole legacy project.

**Cause — not data loss.** The tree is keyed by the file_map mappings: the
331 ported files appear under `converted/` (their target path) and the 9
under `source/` are exactly the `SKIPPED` mappings. The other ~336 legacy
source files are reachable only via their mapping rows — there was no full
source browser.

**Fix.** Added a **Mapping ⇄ Source files** toggle to the Review sidebar:
- Backend: `GET /api/sessions/{id}/source-tree` lists every legacy source
  file (`uploads/<sid>/source/`, `context/` and vendor dirs excluded — 345
  files for `378d02ddd74c`); `GET /api/sessions/{id}/source-file?path=`
  serves one file's text, with path-traversal protection and a 600 KB
  preview cap. ([backend/main.py](backend/main.py))
- Frontend: the **Source files** view renders the full legacy tree
  (reusing `renderTreeNode`); clicking a file opens it read-only,
  line-numbered, in the diff pane.
  ([frontend/review.js](frontend/review.js), [frontend/review.html](frontend/review.html))

**Risk:** low–moderate. The .NET manifest generation is best-effort and
additive — it only writes files when none exist and never overwrites an
agent's own `.csproj`/`package.json`; the worst case is a `dotnet build`
error the user already faced implicitly. The detection change only widens
a fallback. The source-tree endpoints are read-only with traversal guards.
The Review toggle defaults to the existing Mapping view, so the page is
unchanged until the user opts in.

**Sync:** [backend/agents/scaffold.py](backend/agents/scaffold.py),
[backend/agents/run_manager.py](backend/agents/run_manager.py),
[backend/main.py](backend/main.py),
[frontend/review.js](frontend/review.js) and
[frontend/review.html](frontend/review.html) copied to the desktop
checkout along with this changelog. Restart the desktop backend and
hard-reload the Review page for the fix to take effect.

## 3. Low-contrast text — invisible JSON in reports + theme-blind File Map

**Symptom.** In a rendered report, the `file_map.json` code block (and any
JSON/code block) showed string values as near-black-on-black — only
legible once selected with the cursor. The workspace's **File Migration
Map** table used colours that did not adapt to the active theme.

**Root cause — highlight.js theme mismatch.** [frontend/index.html](frontend/index.html)
loaded the **light** highlight.js theme (`styles/github.min.css`), which
colours tokens for a white background — JSON string literals get
`#032f62` (very dark navy). But report code blocks render on a **dark**
background in *both* app themes (`--code-bg` is `#2b2b2a` / `#111110`).
Dark tokens on a dark block → invisible until the selection highlight
puts a light background behind them.

**Fix.** Swapped the stylesheet to the **dark** theme
(`styles/github-dark.min.css`) so token colours are light-on-dark. Added
`pre code.hljs { background: transparent; padding: 0; }` in
[frontend/style.css](frontend/style.css) so the dark theme's own `.hljs`
background doesn't fight the report's `--code-bg`. Bumped the `style.css`
cache-buster to `?v=13`.

**Root cause — File Migration Map was dark-theme-only.** Every colour in
the `.file-map-*` block was hardcoded for a dark surface (`#0f1724`
panel, `#0b1220` header, `#fca5a5` source paths, `#86efac` targets,
`#cbd5e1`/`#94a3b8` body text). On a light surface those light fills and
light text washed out.

**Fix.** Re-pointed the whole `.file-map-*` block at theme variables:
panel → `var(--bg-elev)`, header/inputs → `var(--bg-soft)` / `var(--bg)`,
body text → `var(--text)` / `var(--text-muted)`, and the source/target
path text → `var(--error)` / `var(--success)` — tokens already tuned for
strong contrast on whichever theme surface is active. The summary chips
(`.chip-lite`) likewise moved to `var(--accent-soft)` / `var(--success-bg)`
/ `var(--error-bg)`.

**Risk:** low. CSS-only — a syntax-theme swap plus replacing hardcoded
hex values with the existing theme tokens. No markup or JS change.

**Sync:** [frontend/index.html](frontend/index.html) and
[frontend/style.css](frontend/style.css) copied to the desktop checkout
along with this changelog. Hard-reload the page (Ctrl+F5) so the new
`style.css?v=13` and the dark highlight.js theme are picked up.
