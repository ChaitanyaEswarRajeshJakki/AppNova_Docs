# Changes — 30 June 2026

## fix(scaffold): run frontend dev server in foreground in run.bat (was crashing the dev-server harness)

**What**
- The generated Windows `run.bat` launched the frontend with `start "AppNova Frontend" cmd /k "... npm run dev"` (a detached window) and then `goto :eof`. So the launcher process exited the instant the foreground `npm install` finished — before the frontend port came up. The dev-server harness spawns `cmd /c run.bat` and treats it as the long-running server, so that early exit was reported as `[process_crashed] exited 1` (last output: "installing frontend dependencies ...").
- Changed the frontend launch to run **inline in the foreground** (`cd /d "%FE_DIR%"` + `call npm run dev`), mirroring `run.sh` which already ends on a foreground `npm run dev`. The launcher process now stays alive for the dev server's lifetime; the harness detects `:PORT` listening and keeps the run healthy. Backend still launches in its own visible `cmd /k` window (that part worked — "backend is up").

**Why**
- Pure launcher-structure bug: the app itself is fine (frontend `vite build` succeeds; backend binds `BACKEND_PORT` and comes up). Only `run.bat` diverged from `run.sh` by detaching the foreground server and exiting. Verified `PORT=5050 npm run dev` binds `127.0.0.1:5050` and stays in the foreground.

**Files**
- `backend/agents/scaffold.py` (run.bat template: foreground frontend launch + updated comments)
- `uploads/73d1033e9b9a/converted/run.bat` (applied the same fix to the already-converted app so it runs now)

## fix(converted/TotalBookingAI.API): read `Jwt:Key` instead of nonexistent `Jwt:Secret`

**What**
- `Program.cs` (converted session 73d1033e9b9a) read the JWT signing key from config key `Jwt:Secret` and threw `InvalidOperationException: Jwt:Secret is not configured` at startup. No config file defines `Jwt:Secret` — all three (`appsettings.json`, `appsettings.Development.json`, `appsettings.Production.json`) define `Jwt:Key`. The lookup always returned `null` → app crashed on boot.
- Aligned the code to the config convention: now reads `Jwt:Key`, with an explicit empty/whitespace guard so a blank key fails fast with a clear message instead of producing an unusable signing key.

**Why**
- Single name mismatch between the only JWT consumer (`Program.cs`) and the config. Fixing the code (one site) rather than renaming the key across three config files preserves the production deploy substitution placeholder `__JWT_SIGNING_KEY__` under `Jwt:Key`. There is no token-generation code (auth uses a DB `authKey` via `FindByAuthTokenAsync`), so `Program.cs` is the sole reader — no second site to keep in sync.

## fix(scaffold): never emit `cd backend && dotnet run` for a nested-csproj split

**What**
- Hardened the run-script generator in `backend/agents/scaffold.py` so a backend whose runnable `.csproj` sits *below* `backend/` (e.g. `backend/<Name>.API/<Name>.API.csproj`) can never get a run script that `cd`s into `backend/` — where only a `.sln` lives — and fails with **"Couldn't find a project to run."**
- New helpers:
  - `_is_test_csproj()` — excludes test projects (name suffix `.Tests`/`.Test`/… or a test-SDK/xunit/nunit/mstest package reference) from API selection.
  - `_csproj_is_web()` — identifies an ASP.NET app via the `Microsoft.NET.Sdk.Web` attribute or web fingerprints (`Program.cs` / `Controllers/` / `appsettings*.json`).
  - `_resolve_runnable_api_dir(root)` — robust, deterministic pick of the runnable project dir: ranks Web+`Program.cs` > `Program.cs` > Web > first non-test project; returns `None` only when no non-test `.csproj` exists.
- Wired the resolver into the three spots that previously produced a `None`/`"backend"` path:
  1. `_ensure_dotnet_manifests` early-return (csproj already present) now resolves via `_resolve_runnable_api_dir` instead of a `Program.cs`-sibling-only loop that returned `None` for a Web project without an adjacent `Program.cs`.
  2. `_ensure_dotnet_manifests` generation branch falls back to the resolver when its `Program.cs` scan didn't pin an API dir.
  3. `_write_split_run_scripts` self-resolves the API dir when the caller passes `api_project_dir=None`, and only lands on the literal `"backend"` when there is genuinely no runnable project to point at. The `relative_to` now resolves both sides so an 8.3/symlinked `converted_dir` can't spuriously fall back to `"backend"`.

**Why**
- Real conversion (session 73d1033e9b9a) shipped `run.bat`/`run.sh` that ran `dotnet run` from `converted/backend/`, which held only `Backend.sln` + nested `TotalBookingAI.API/`. The scripts were `_write_split_run_scripts` output emitted with `api_project_dir=None`, so the emitter fell back to `cd backend` — exactly the failure path. This makes the generator layout-correct at the source instead of relying on the `ensure_mandatory_structure` backstop to rewrite stale scripts after the fact.

**Validation**
- `python` harness over the real tree + synthetic layouts: real nested layout → `backend/TotalBookingAI.API`; flat backend (csproj directly in `backend/`) → `backend`; nested API+Tests+`.sln` → `backend\My.API` (bat) / `backend/My.API` (sh) with the Tests project excluded; `_ensure_dotnet_manifests` early-return resolves `My.API`; no-csproj → `None` → `backend` fallback. All pass.
- The already-broken converted session was also repaired in place (its `run.bat`/`run.sh` now point at `backend/TotalBookingAI.API`).

## fix(codegen): add deterministic post-codegen structural validation pass

**What**
- New `backend/agents/codegen_validators.py` — a deterministic (no-LLM) validator that runs after every code-generation pass and writes `docs/CODEGEN_VALIDATION.md`. Four checks, each targeting a build-blocking defect class observed in a real conversion (session 73d1033e9b9a):
  1. **`.csproj` layout** — flags a project file that is an ancestor directory of another `.csproj`, because SDK-style globbing then compiles the nested project's sources (e.g. test sources pulled into the web build → "Xunit/Moq could not be found").
  2. **namespace == type collision** — flags a type whose name equals its namespace's leaf segment (e.g. `class Agency` in `...Entities.Agency`), the cause of CS0118 "is a namespace but is used like a type".
  3. **orphan `using`** — flags `using X.Y.Z;` under the project's own root namespace that no file declares (CS0234), scoped to internal roots so framework/package namespaces are never false-flagged.
  4. **placeholder residue** — flags deploy-template tokens like `__APP_DISPLAY_NAME__` that leaked into C# source as bare identifiers; placeholders inside string literals are excluded (those are the deploy quarantine pass's deliberate config sentinels).
- Wired into `backend/agents/supervisor.py`: import added, and a guarded call after the existing deploy-audit block stashes the result on `result["codegen_validation"]`. Advisory only — it does not compile the project or downgrade status by itself.

**Why**
- A recent conversion shipped a backend that failed `dotnet build` immediately: `__APP_DISPLAY_NAME__` leaked into 137 namespaces, an `Agency` namespace/type collision, seeders importing a non-existent `.Models` namespace, and the API `.csproj` written to `backend/backend.csproj` (parent of the API + test projects) so the web build swallowed the test sources. A previous conversion didn't hit these edges, so nothing caught them. These are exactly the structural classes per-file LLM generation can't see; a cheap deterministic sweep catches them before the output is called done.

**Validation**
- Ran against the (now-repaired) converted tree: 168 .cs files scanned, **0** false positives on the fixed checks; correctly still flags the latent `TotalBooking` namespace/type collision (builds today only because every consumer fully-qualifies `TotalBooking.TotalBooking`) and correctly does NOT flag the in-string `__SSO_ALLOWED_DOMAINS__` quarantine sentinel.
- `python -m py_compile` clean on both files.

**Files**
- `backend/agents/codegen_validators.py` (new)
- `backend/agents/supervisor.py` (import + post-codegen call)
