## fix(scaffold): split-stack run.bat must keep backend in-process so AppNova can see it

Repro: clicking "Run Converted" on session `60f92f3e34b5` in the AppNova hub streamed the frontend log into the run card (Vite ready, port 5050) but no backend window ever opened and the converted app rendered blank in the browser because every `/api` call 502'd through Vite's proxy.

### Root cause

The previous `run.bat` template emitted by `scaffold._write_split_run_scripts` used `start "AppNova backend" cmd /c "..."` to spawn the dotnet process in a SEPARATE console window. That works when the user double-clicks `run.bat` from Explorer — they get two visible windows — but it does NOT work when AppNova's run-manager spawns the script via `subprocess.Popen` with stdout/stderr piped into the hub log stream. The parent cmd in that case has no interactive console attached, so:

- The detached `start` either silently fails to allocate a window OR opens one that's immediately hidden behind every other top-level window. The user sees nothing.
- If `dotnet run` errors inside that invisible window (missing csproj, port in use, EF migration failure, anything), there is no surface anywhere — not in the hub log, not on the user's desktop, not in any file — for them to read the failure from.
- `run.bat` then `echo`s "[AppNova] frontend (node)" and proceeds to `npm run dev`. Vite starts cleanly because the JS side doesn't care that the backend isn't there yet. The page loads, every `/api` call 502s through the proxy, and the SPA renders whatever its bootstrap-time errors produce — typically blank.

The bash equivalent (`run.sh`) was already correct: it backgrounds dotnet with `&`, captures `$!` for cleanup, polls `127.0.0.1:$BACKEND_PORT` in a `for i in $(seq 1 60)` loop until the port answers, then starts Vite. Windows just never got the same treatment.

### Fix — `backend/agents/scaffold.py:_write_split_run_scripts`

Rewrote the Windows half. The new `run.bat`:

1. `del /Q backend.log 2>nul` — clear any stale log from a prior failed run.
2. `start /B "" cmd /c "cd /d <api> && set ASPNETCORE_URLS=... && dotnet run > "%~dp0backend.log" 2>&1"` — `/B` keeps the dotnet process in the SAME console + process group as the parent, suppresses the new-window allocation entirely, and `> backend.log 2>&1` writes everything dotnet emits to a file next to `run.bat`. AppNova's run-manager (which already tails files for the converted-output preview) can now stream that file into the hub UI, and if dotnet crashes the user has a real on-disk log to read.
3. PowerShell polls `127.0.0.1:%BACKEND_PORT%` via `Net.Sockets.TcpClient.Connect`, retrying every second for up to 60 seconds. `errorlevel 1` from the powershell call means the backend never came up — in that case the script `type backend.log`s its content into the parent stdout (which IS streamed to the hub) so the user sees the dotnet error inline, then continues to start the frontend anyway. The frontend coming up alone is still useful for diagnosis (you can hit the Vite dev page even if the backend is dead).
4. The frontend half is unchanged from the prior template — `npm run dev --port %PORT% --host 127.0.0.1` in the foreground, stdout piped to the parent.

The bash `run.sh` template is untouched.

### Fix — `backend/agents/scaffold.py:ensure_build_manifests`

Adjacent bug: `ensure_build_manifests` would only call `_write_split_run_scripts` when `created` was non-empty AND the layout was split. That meant deleting a stale `run.bat` and re-running the scaffold against a session whose manifests were otherwise complete was a no-op — the broken script wouldn't be regenerated. Now also re-emits the run scripts when EITHER `run.sh` or `run.bat` is missing, regardless of whether anything else needed backfilling. Lets the user delete one of the scripts and re-run the scaffold to get a fresh copy.

```python
run_sh_missing = is_split and not (converted_dir / "run.sh").exists()
run_bat_missing = is_split and not (converted_dir / "run.bat").exists()
if is_split and (created or run_sh_missing or run_bat_missing):
    created += _write_split_run_scripts(...)
```

## fix(prompts,deploy_audit): ban `__FOO__` placeholders inside compiled source

Repro: re-ran "Run Converted" on `60f92f3e34b5` after the run.bat fix above. Backend.log now surfaces the real failure — `dotnet build` rejects three repository files with `CS1003` / `CS1002` because the generated C# contains the literal token `__DB_ADMIN_USER__`:

- `backend/src/TotalBooking.Infrastructure/Repositories/PcDeclarationRepository.cs:269,523` — inside an object initializer where `UserId = userId,` belonged.
- `…/TotalBookingRepository.cs:171` — same shape.
- `…/UserRepository.cs:160` — spliced into the middle of an expression: `current__DB_ADMIN_USER__ id` instead of `currentUserId == id`.
- `…/UserRepository.cs:193` — `new SignatureAudit { __DB_ADMIN_USER__ Signature = ... }`.

Backend never binds `:5052`, so run.bat's 60-second port poll times out and the run card flips to `process_crashed`. April session worked because it pre-dates the codegen prompt expansion in `a9a4a96`; every May conversion (`60f92f3e34b5`, `c9afaa02f85e`) reproduces the leak.

### Root cause (placeholder leak)

[backend/agents/prompts.py:248-258](backend/agents/prompts.py#L248) teaches the codegen model that when a `deploy_config` field is blank, it should emit a canonical `__FIELDNAME__` placeholder and the deploy_audit allowlist will treat it as clean. The examples in that section all live in `appsettings.Production.json` connection strings, but the model over-generalises and reaches for the same fallback when it doesn't know what to write in C# repository code.

The deploy_audit pass at [backend/agents/deploy_audit.py:710-738](backend/agents/deploy_audit.py#L710) lists `__DB_ADMIN_USER__` (and friends) on `_PLACEHOLDER_DESCRIPTIONS` with no file-extension guard, so the placeholder allowlist silently blesses the leak everywhere it appears — including inside `.cs` files where the compiler can't parse it.

### Fix — `backend/agents/prompts.py` (codegen section)

Inserted a new top-level rule block right after the placeholder examples and before "Banned-source policy":

> **⚠️ PLACEHOLDERS ARE FOR CONFIG FILES ONLY — NEVER FOR SOURCE CODE**
>
> Allowed homes: `appsettings*.json`, `.env*`, `azure-pipelines*.yml`, `.github/workflows/*.yml`, `*.bicep`, `*.tf`, `*.tfvars`, `devops/**`, `README*.md`, `DEPLOYMENT.md`, `DATA_MIGRATION.md`, `docs/SECRETS_MAPPING.md`.
>
> Hard ban — a single `__[A-Z_]+__` token in any of these source extensions FAILS THE BUILD and triggers a quarantine retry: `.cs`, `.ts`, `.tsx`, `.js`, `.jsx`, `.mjs`, `.cjs`, `.py`, `.go`, `.java`, `.kt`, `.scala`, `.rb`, `.vue`, `.svelte`, `.php`, `.sql`, `.cshtml`, `.razor`, `.html`.

And a "when you genuinely don't know a value in source code" fallback table — typed default + `// MUST-FIX:` comment per language, so the reviewer can grep `MUST-FIX:` to enumerate every gap. Crucial: a placeholder in source is a silent build break; a `// MUST-FIX:` is a known-cost gap.

### Fix — `backend/agents/deploy_audit.py`

Added Phase 8 — placeholder-in-source detection:

1. `_SOURCE_CODE_EXTS` — the compiled / type-checked extensions where any `__FOO__` token is illegal. Excludes JSON / YAML / Markdown / IaC / CI / shell / CSS so the existing placeholder convention in config files is untouched.
2. `_PLACEHOLDER_LIKE_RE = re.compile(r"\b__[A-Z][A-Z0-9_]{2,}__\b")` — matches `__UPPER_SNAKE__` shapes; the case + length guard skips Python dunders (`__init__`, `__name__`) and stylistic `__X__` markers.
3. `_PLACEHOLDER_ALLOWLIST_IN_SOURCE` — explicit pass for build-time defines that legitimately wear the same costume (`__DEBUG__`, `__DEV__`, `__VERSION__`, etc. from Vite / webpack).
4. `_scan_file_for_misplaced_placeholders(path, rel)` — runs only on source-code extensions; emits `LeakHit(field="placeholder_in_source", literal="<the bad token>")` per occurrence. Wired into `audit_deploy`'s per-file loop alongside the existing leak scanners.
5. `quarantine_leaks` skips `placeholder_in_source` hits — auto-rewriting a `__DB_ADMIN_USER__` to `__PLACEHOLDER_IN_SOURCE__` would only deepen the build break. Leaving the literal means the supervisor's `issue_count` gate still counts it and `DEPLOY_AUDIT.md` surfaces the file + line for the reviewer.

### What this catches that nothing else did

The existing `_LEAK_TO_FIELD` and `_LEAK_PATTERNS` scanners run per-line but feed every match through `_is_placeholder`, which matches any `^__[A-Z0-9_]+__$` and silences the hit. That allowlist made structural sense for `appsettings.Production.json`, but it also blinded the auditor to the exact failure mode we're now seeing in C#. The new Phase 8 scanner intentionally bypasses `_is_placeholder` because the whole point is to flag a placeholder that's been written into the wrong kind of file.

### Refinement after regression-checking against the April working session

Before declaring victory I scanned the working April session `378d02ddd74c` (552 files) with the new Phase 8 scanner — and hit two categories of false positives that would have downgraded a session that runs fine:

- `node_modules/**` — vendor JS uses `__PURE__`, `__REACT_DEVTOOLS_GLOBAL_HOOK__`, `__REDUX_DEVTOOLS_EXTENSION__`, `@@__IMMUTABLE_*__@@` etc. These are already pruned by `_walk_converted`'s `_SKIP_DIRS`. **Safe by construction.**
- Real source code in `378d02ddd74c/backend/src/ARIES.API/Program.cs:34` and `…/AppConstants.cs:44`:

  ```csharp
  builder.Configuration.GetConnectionString("Default") ?? "__DB_CONNECTION_STRING__"
  public const string DevEmail = "dev@__SSO_ALLOWED_DOMAINS__";
  ```

  These compile fine — the placeholder lives inside a string literal as a deliberate "let the operator see the missing-config name at runtime" marker. **Not a build break, not a bug.**

The CRITICAL distinguisher between the bug shape and the legit shape is whether the token is a **bare identifier** (compile error) or **inside a string / comment** (compiles cleanly):

| Shape | Example | Compiles? |
| --- | --- | --- |
| Bare identifier | `UserId = __DB_ADMIN_USER__,` | No — CS1003 |
| Spliced identifier | `current__DB_ADMIN_USER__ id` | No — CS1002 |
| Inside string literal | `?? "__DB_CONNECTION_STRING__"` | Yes — runtime marker |
| Inside line comment | `// MUST-FIX: replace __DB_ADMIN_USER__` | Yes |

Two refinements to the Phase 8 scanner so it only flags the build-breakers:

1. **`_position_is_in_string_or_comment(line, pos)` helper.** Walks the prefix of the line, toggling `in_double` / `in_single` / `in_backtick` on unescaped quote characters; tracks `//` / `#` / `--` line-comment markers OUTSIDE string state. Handles C# `@"…"` / `$"…"` (the leading sigil doesn't affect quote toggling), Python triple-quoted bodies (three independent toggles leave us correctly inside), and `\"` escapes. Multi-line `/* */` comments are NOT modelled — but the legit pattern is always string-literal, so this gap doesn't matter in practice. The scanner calls this helper before emitting a hit; in-string and in-comment matches are skipped.

2. **Dropped the `\b` word boundaries from `_PLACEHOLDER_LIKE_RE`.** The previous regex `\b__[A-Z][A-Z0-9_]{2,}__\b` missed the WORST form of the bug — the spliced variant `current__DB_ADMIN_USER__ id` at UserRepository.cs:160 — because `\b` doesn't fire between two word-chars like `t__`. The dunder false-positive concern (`__init__`, `__name__`) is already handled by the leading-uppercase guard `[A-Z]`, so the boundaries weren't pulling weight. New regex: `__[A-Z][A-Z0-9_]{2,}__`.

## feat(supervisor): code-review + ui-ux now drive the live converted app in a bounded fix→retest loop

Request from session: *"from now on the after code generation code review agent should run browser tests itself for both source and converted app and find all the issues while it should run in backend and also make fixes and write to converted files respectively and then ui x agent see through the ui of the converted app and if it is not able to see the browser and ux is not rendering it should also fix the issues and move on to next sections for approval and marking for running"*

User-confirmed scope (via three-question intake):

1. **Source vs converted**: Converted-only browser test. Skipping source-side comparison — launching a PHP/Laravel/AngularJS/Java legacy stack autonomously is a multi-week per-stack effort, brittle, and orthogonal to the user's actual pain (converted apps that don't boot or render blank).
2. **Iteration budget**: Up to 3 fix→retest passes per agent. Hard cap bounds token cost; remaining issues land in the report for human review.
3. **Approval gate**: Auto-flip session to `ready_to_run` when ui-ux completes with 0 P0/P1 issues. User still clicks the Run button — they're just told up-front it's ready.

### Before this change

- `code-review` ran prompt-only against the static source/converted file tree. Could not catch: backend crash on boot, blank-page renders, console errors, 4xx/5xx on form submit, missing imports, broken `<Route>` paths — anything that only surfaces when the app actually runs.
- `ui-ux` referenced screenshot filenames in its markdown output but had no way to actually SEE the rendered page. The agent emitted `![Legacy](...)![Converted](...)` syntax that the user read; the agent itself was guessing.
- No supervisor-driven browser test existed. `backend/agents/browser_test.py` ran only when the user clicked the Browser Test card after a successful Run Converted; the post-codegen agents never benefited from its findings.
- No auto-promotion to ready_to_run. The "Marked Ready by admin" badge was always a manual click via [`/api/review/{sid}/ready-to-run`](backend/main.py#L1976).

### Fix — new module [backend/agents/post_codegen_browser_loop.py](backend/agents/post_codegen_browser_loop.py)

- `gather_browser_evidence(session_id, converted_dir, screenshots_dir)` — spins up `run.bat`/`run.sh` via [run_manager.start_run](backend/agents/run_manager.py#L952), waits for the dev server, calls [browser_test.run_browser_test](backend/agents/browser_test.py#L162) at the allocated port, stops the run, returns the manifest. Try/finally guarantees `stop_run` runs even if Playwright raises; on any `start_run` / `browser_test` failure returns a synthetic manifest with `ok=False` + `failure_kind` so the caller can surface the failure to the agent as evidence (instead of silently degrading to prompt-only).
- `format_evidence_for_prompt(manifest, *, converted_dir, with_screenshots, pass_number, max_passes)` — builds the markdown block appended to the agent's system prompt. Three branches:
  1. `ok=False` → "❌ Browser test could not run" with the failure_kind and a directive to read `backend.log` and fix the boot.
  2. `suggested_fixes` non-empty → enumerated issue list with `detail` + `hint` per item.
  3. Clean run → "✓ No runtime issues" with a directive NOT to introduce new edits without evidence.
  Plus: deduped + prioritized log entries (page_error > 5xx > request_failed > console_error > 4xx > console_warning), visited-page list, optional screenshot list.
- `with_screenshots=True` (ui-ux only) emits explicit "READ ALL of these" lines pointing at the PNG file paths. The Claude Code CLI's `Read` tool natively handles PNG/JPG as multimodal input — listing the paths IS the vision channel. No SDK refactor needed.
- `loop_enabled()` reads `APPNOVA_BROWSER_LOOP_ENABLED` (default `true`). Set `false` to revert to pre-loop behaviour without redeploying.

### Fix — supervisor wiring [backend/agents/supervisor.py:1228](backend/agents/supervisor.py#L1228)

Wrapped the existing `_invoke(full_prompt)` call site with a loop that activates only when `agent_id in {"code-review", "ui-ux"}` AND `converted_dir is not None` AND `loop_enabled()`. Per-pass behaviour:

1. **Gather** evidence via `gather_browser_evidence`. On harness fault, fall through to a synthetic evidence dict so the agent runs at least once with the failure visible.
2. **First-pass early-exit**: if pass 1 returns a clean run (no `suggested_fixes`), skip the agent invocation entirely and break — the converted app is healthy and the prompt-only review would waste budget. The baseline `_invoke(full_prompt)` is still called after the loop to produce the report.
3. **Inject + invoke**: append `format_evidence_for_prompt(...)` to the prompt, run the agent with the cwd-lock held (same write serialization as the pre-loop path), record the pass result.
4. **Stop on agent error**: if the agent returns `status != "done"` (timeout, runtime crash), break — don't keep spending budget against a broken invocation.
5. **Stop at cap**: at pass `_BROWSER_LOOP_MAX_PASSES` (3), break with `reason="budget exhausted"`.
6. **Final probe** (only when a fix loop actually ran): re-launch + browser-test ONE more time after the last agent edit so the residual issue count is recorded as `_browser_loop.final_issue_count`.

The aggregated outcome attaches to the agent's result dict under `_browser_loop` (passes_run, final_clean, final_issue_count, reason, per_pass[]) so the wave summariser and the auto-ready gate downstream can both consume it without re-running anything.

### Fix — auto-promote to ready_to_run [backend/agents/supervisor.py end of run_supervised](backend/agents/supervisor.py#L2485)

After the wave loop completes, inspect `ui-ux` and `code-review` results:

- ui-ux must have `status == "done"` AND `_browser_loop.final_clean == True` AND `_browser_loop.final_issue_count == 0`.
- code-review must either be absent (excluded from this run) OR have `status == "done"`. If code-review ran the browser loop and didn't end clean, it BLOCKS the auto-flip — we don't promote over the top of unresolved code-review hits.

When the gate passes, late-bound import of `backend.review` (avoids circular dep — supervisor is a leaf today, keeping it that way) calls [`set_ready_to_run(session_id, by="supervisor:auto", note=...)`](backend/review.py#L433) with a note describing how many passes each agent ran. Failure modes (FileNotFoundError if SessionReview isn't on disk yet, etc.) are caught and logged — the supervisor's return value is never affected. On success, an `auto_ready` progress event fires so the SSE-subscribed hub UI can flip the badge in real time.

### Fix — prompts: tell each agent how to consume the evidence

- [prompts.py code-review section](backend/agents/prompts.py#L1216): added "⚠ Live browser-test evidence is appended at the very end of this prompt" block after the goal line. Tells the agent to scroll to the bottom first, treat live-evidence issues as strictly higher priority than static-analysis findings, and orients them on the up-to-3-pass loop semantics.
- [prompts.py ui-ux section](backend/agents/prompts.py#L1568): added "⚠ You have REAL VISION input — read it first" block. Documents that the screenshot paths in the evidence block are real multimodal Read targets, lists the visual judgements the agent should make (layout match, wizard rendering, blank rectangles, runtime-error overlays), and clarifies that live screenshots take priority over the static legacy renders for fidelity comparison.

### Why this design (not the alternatives)

- **Why not switch ui-ux to the Anthropic SDK for image_block input?** The Claude Code CLI's `Read` tool already presents PNG/JPG content as multimodal input. Listing paths in the prompt IS the vision channel — no SDK refactor, no new auth path, no parallel runner. Single-source-of-truth keeps the runner.py contract intact.
- **Why a bounded loop instead of "iterate until clean"?** User explicitly chose 3 passes. A converging fix typically lands in pass 1; a converging-but-noisy fix in pass 2; pass 3 is the safety. Beyond 3 the marginal value drops fast and tokens compound — better to surface residual issues to the human reviewer than to burn the session budget chasing them.
- **Why not auto-launch the source app for parity comparison?** Source stacks vary wildly per session (PHP/Laravel/MySQL, AngularJS/Express/Mongo, Java/Spring/Postgres, etc.). A general-purpose source launcher is a multi-week per-stack effort (composer, npm, db migrations, vhost config). User opted out — converted-only browser test ships in this session and covers the actual pain.
- **Why not auto-trigger Run Converted after auto-ready?** User opted for "auto-mark Ready, user still clicks Run." Conservative — leaves the human in the loop for the final go/no-go without making them gate the agent work.

## fix(dev_assist): per-file fallback for apply_diff when a single hunk breaks a multi-file patch

Repro: tried to fix the `__DB_ADMIN_USER__` leak above through the dev-chat. The model proposed a 3-file diff (`PcDeclarationRepository.cs`, `TotalBookingRepository.cs`, `UserRepository.cs`). The Apply button surfaced an alert:

> `error: patch failed: backend/src/TotalBooking.Infrastructure/Repositories/PcDeclarationRepository.cs:520`
> `error: backend/src/TotalBooking.Infrastructure/Repositories/PcDeclarationRepository.cs: patch does not apply`

Nothing landed — even though the `TotalBookingRepository.cs` hunk shown in the preview was textually correct and would have applied cleanly on its own.

### Root cause (apply_diff all-or-nothing)

[backend/agents/dev_assist.py:744 apply_diff](backend/agents/dev_assist.py#L744) hands the WHOLE diff to `git apply --recount --ignore-whitespace`. git apply's contract is all-or-nothing: a single hunk whose context lines drift (extra whitespace, off-by-one position, CRLF nuance the LF-normalize step didn't catch) rejects every hunk in the patch. The dev-chat prompt at [prompts.py:102-106](backend/agents/prompts.py#L102) already warns the model about this, but a 3-file diff against a 50KB+ tree still drifts often enough that the all-or-nothing policy was leaving the reviewer with nothing actionable.

### Fix — `backend/agents/dev_assist.py`

Added `_split_diff_by_file(diff_body) -> list[(path, sub_body)]` that walks the diff and slices at every `diff --git a/X b/X` boundary (or bare `--- a/X` when no `diff --git` header is present). Index / mode / new-file headers between `diff --git` and `---` are carried into the slice as-is.

`apply_diff` now does this on a full-patch failure (forward applies only — `--reverse` keeps the all-or-nothing behaviour because block-order matters in pathological cases for reverse):

1. Whole-patch `git apply` runs first (fast path; succeeds for clean diffs).
2. On non-zero exit AND `len(sub_diffs) > 1`, retries each per-file slice via a recursive `apply_diff` call.
3. Aggregates results:
   - **All slices succeed** → return `ok=True` with the merged file list. (This actually happens — git's `--recount` at file boundaries occasionally over-eats context lines from the next block and rejects a multi-file patch even though each block alone is fine.)
   - **Mixed** → return `ok=False, applied=True, files=<applied>, failed_files=[{path, stderr}, ...]` with a human-readable `stderr` summary (example below).
   - **All fail** → same shape with `applied=False`.

Example `stderr` on partial apply:

```text
2 of 3 file(s) applied; 1 failed.
Applied:
  • backend/.../TotalBookingRepository.cs
  • backend/.../UserRepository.cs
Failed (regenerate these in dev-chat):
  • backend/.../PcDeclarationRepository.cs: error: patch failed at line 520
```

### Caller behaviour preserved

[backend/main.py:6631](backend/main.py#L6631) still requires BOTH `applied` AND `ok` to mark a diff as applied in the persisted thread. A partial apply (`applied=True, ok=False`) does NOT auto-mark and does NOT auto-restart the run — both correct, because the diff isn't fully landed. The reviewer sees the summary, knows which files now sit modified on disk, and can ask the dev-chat to regenerate just the failing file in the next turn.

### Why not `git apply --reject`

`--reject` writes `.rej` files next to the patched targets. On the converted/ tree those would either (a) get picked up by file-watcher builds that don't ignore `.rej` (Vite glob patterns vary by template), or (b) confuse the reviewer flipping between Preview / Apply rounds. Splitting the patch produces the same partial-apply effect without leaving litter in the working tree.

## feat(scaffold): converted-app run.bat opens visible backend + frontend log windows alongside the hub stream

Symptom: after the 2026-05-26 `fix(scaffold): split-stack run.bat must keep backend in-process` commit, clicking Run Converted no longer popped visible console windows the way the pre-fix scaffold did. The hub still got its log stream, but the user lost the ability to glance at a separate desktop window for backend or frontend output. The earlier fix had to drop visible windows because `start "Title" cmd /c <cmd>` either silently failed or opened a hidden window when AppNova's run-manager spawned `run.bat` via subprocess.Popen.

### Solution — `backend/agents/scaffold.py:_write_split_run_scripts`

Keep the in-process `/B` launch + `> backend.log 2>&1` redirect (so the hub still tails the log into the run card), AND additionally spawn one `start "Title" powershell -NoExit` window per stack that runs `Get-Content -Path <log> -Wait -Tail 200`. Result:

- **Backend log window**: `start "AppNova: backend log" powershell -NoExit -NoProfile -Command "Get-Content -Path 'backend.log' -Wait -Tail 200 -ErrorAction SilentlyContinue"` opens immediately after the `del /Q backend.log` + `type nul > backend.log` lines that ensure the file exists. The dotnet `/B` child writes to backend.log; PowerShell streams new lines into the window in real-time.
- **Frontend log window**: same pattern with `frontend.log`. The npm run line is now wrapped in PowerShell + `Tee-Object -FilePath '%~dp0frontend.log'` so the hub's parent-cmd-stdout stream still sees Vite output (Tee writes to BOTH stdout and the file) while the visible window tails the file.

### Why `Tee-Object` and not a raw `>` redirect

A plain `> frontend.log 2>&1` would point npm's stdout at the file ONLY — the hub's run card preview pane would go dark because the parent cmd would have nothing to stream. `Tee-Object` fans the same output to both the file AND the PowerShell pipeline, which inherits the parent cmd's stdout.

### Why `type nul > backend.log` before starting the tail window

PowerShell's `Get-Content -Wait` errors if the file doesn't exist at the moment it's called. Touching the file first guarantees the window opens cleanly. `-ErrorAction SilentlyContinue` is a defensive belt-and-suspenders against rare midflight race deletions.

### Why `-NoExit`

The PowerShell window stays open after the dotnet/npm process ends so the user can scroll back through the final crash trace. Without it, a failed `dotnet run` would close the window the instant `Get-Content` saw the file stop growing — exactly when the user needs the log most.

### Bash `run.sh` — unchanged

Linux/macOS users have `tail -f backend.log` ergonomics they already know; opening a new terminal app from a shell script is implementation-defined (gnome-terminal vs xterm vs Terminal.app vs WSL conhost) and not worth the brittleness. Skipped the symmetric change there.

## fix(browser-test): smart-tail log extraction — find errors anywhere in the file, not just the last 200 lines

Initial concern from the user: "what if the error is at line 500 or line 10000?" The previous `_extract_error_context` was a naive last-200-lines tail. A 10,000-line `backend.log` where the unhandled exception fires at line 500 (followed by 9500 lines of routine request logging from the still-running process) would have surfaced lines 9800-10000 — none of which contain the actual exception text.

### Fix — `backend/agents/browser_test.py:_extract_error_context`

New helper that scans the WHOLE log file for error-shaped lines, captures a context window around each match, merges overlapping windows, and ALSO appends the end-of-log tail as a silent-crash safety net. Output preserves original line numbers as `NNNNNN:` prefixes (followed by a space) so the chat agent can grep the source log if it needs more context. Windows separated by gaps emit a `... (N lines skipped) ...` marker between them.

#### Error patterns scanned

```python
_LOG_ERROR_PATTERNS = (
    # .NET runtime / startup
    r"(?i)\bunhandled\b.*exception",
    r"^\s*\w+(\.\w+)+Exception\b",       # System.InvalidOperationException
    r"^\s+at \S+\.\S+\(",                 # C# stack frame
    r"\bSystem\.\w+Exception\b",
    r"(?i)\bstack ?trace:",
    # MSBuild + C# compile
    r"\berror MSB\d{4}\b",
    r"\berror CS\d{4}\b",
    r"\bMSBUILD : error\b",
    r"Build FAILED\.?",
    r"\bRestore failed\b",
    # TypeScript / Vite / Node
    r"\berror TS\d{4}\b",
    r"^\[vite\] (Pre-transform error|Internal server error|Failed to load)",
    r"(?i)\bFAIL\b\s+\S+",               # vitest/jest FAIL banner
    r"^(EADDRINUSE|ECONNREFUSED|ENOENT|EACCES)\b",
    r"^npm ERR!",
    # Generic
    r"(?i)^\s*(?:traceback|fatal|panic):",
    r"(?i)^error:",
)
```

#### Tunables

```python
_LOG_CONTEXT_BEFORE = 5         # lines of pre-context per match
_LOG_CONTEXT_AFTER  = 30        # lines after (enough for a typical C# stack trace)
_LOG_END_TAIL_LINES = 50        # always-include trailing window
_LOG_EXTRACT_MAX_BYTES = 8 * 1024  # hard cap per file so chat context isn't blown
```

#### Behaviour on different inputs

| Input | Output |
| --- | --- |
| 10,000-line log, exception at line 500, npm err at line 9950 | Lines 495–532 (around exception) + `... (9412 lines skipped) ...` + lines 9945–10000 (npm err + tail). 4 hits, ~4 KB |
| 300-line log, no error patterns at all | Last 50 lines only (silent-crash fallback). 0 hits |
| Empty log | `None` |
| 3-line log with error at line 1 | Whole file. 1 hit |

#### Render-report metadata

The browser-test markdown report headings now show match counts and total log size:

```markdown
## Backend stack trace (backend.log — 4 error pattern(s) found across 10000 line(s))
```

So the user knows at a glance whether the extraction is comprehensive vs a possible miss.

- Synthetic 10k-line log with `Unhandled exception` at line 500 + `npm ERR!` at line 9950: both surface in the extraction. Confirmed via REPL.
- Empty log + clean (no-error) log + short (3-line) log all handled correctly per the table above.
- `python -c "import ast,io; ast.parse(io.open('backend/agents/browser_test.py',encoding='utf-8').read())"` passes.

## fix(scaffold): rewrite split-stack launcher — cmd-only run.bat + _run_backend.cmd helper, port pre-clear, drop --port hack

After a long iteration today trying to get Run Converted working on session `378d02ddd74c` (which has the nested `backend/src/ARIES.API/` layout), we kept hitting one launcher bug after another:

1. Original template emitted `_dual_stack_run_script` which did `cd backend && dotnet run` — failed because the csproj is at `backend/src/ARIES.API/`, not `backend/`.
2. `_write_split_run_scripts` correctly resolved `api_project_dir` but the cmd `start /B "" cmd /c "cd /d ""%~dp0..."" && set ... && dotnet run > ..."` chain had nested-quote escaping that broke cmd's parser depending on what was wrapping it.
3. Tried a PowerShell run.ps1 thin-wrapper approach — worked end-to-end (backend up, Vite ready) BUT PowerShell consumed the `--` separator in `npm run dev -- --port $PORT --host 127.0.0.1`, leaving Vite to bind to its default `5173` instead of the run-manager's allocated `5050` → run-manager declared `port_never_bound`.
4. User asked to revert to cmd (no PowerShell wrappers).

Meanwhile, a fully-working April-era session (`c9afaa02f85e`) was sitting in `uploads/` and had been launching cleanly via its own `run.bat`. Read it, distilled the patterns that worked, and rebuilt the May template around them.

### Rewrite — `backend/agents/scaffold.py:_write_split_run_scripts`

Three files emitted now instead of two:

- **`run.bat`** — pure cmd launcher. Defensive `freeport` (from April pattern). Spawns dotnet via `start /B "" _run_backend.cmd`. Polls TCP via `netstat | findstr` (no PowerShell). Runs npm in the foreground so the run-manager pipes stdout to the hub. Drops `--port`/`--host` CLI flags because the scaffold-emitted `vite.config.ts` already reads them from env.
- **`_run_backend.cmd`** — tiny helper. `cd %BE_API_DIR%` + `set ASPNETCORE_URLS=...` + `dotnet run > %BE_LOG% 2>&1`. Having this in a separate file means run.bat doesn't have to wrestle with `cmd /c "..."` nested-quote escaping — cmd's parser is fragile when multiple quoted strings concatenate inside `/c`.
- **`run.sh`** — bash equivalent. `freeport()` helper using `lsof -ti`. Backgrounds dotnet directly with `&` (no helper needed because shell escaping is sane). Same TCP poll + foreground frontend pattern.

### What got borrowed from the April c9afaa02f85e session

| Pattern | Why it's worth porting |
| --- | --- |
| `call :freeport <port>` pre-clear | Stale processes from prior crashed runs hold ports. The hub allocates fresh ports per run from its pool, but a long-running stuck dotnet from a manual test can still squat on a fixed port the user expected to be free. April's template killed them defensively. |
| `netstat`-piped-to-`findstr` LISTENING poll | Pure cmd, no PowerShell dependency. Faster startup (no PS process spawn). Matches what the April template used. |
| Helper batch (`_run_backend.cmd`) instead of inline `cmd /c "..."` | Cmd's quoting rules in nested `cmd /c "..."` contexts are notoriously fragile — different cmd versions strip outer quotes inconsistently, and any internal `"` for a `set "VAR=value"` can break the chain. Separating the work into its own .cmd file makes the quoting trivial. |

### What got dropped (no longer needed)

- The `powershell ... Start-Process ...` tail-window calls. They worked from PowerShell parents but not from cmd parents under piped Popen, and added another moving part with another quoting layer.
- `--port %PORT% --host 127.0.0.1` on `npm run dev`. Vite reads these from env via the scaffold-emitted `vite.config.ts` (already at [backend/agents/scaffold.py:836-857](backend/agents/scaffold.py#L836-L857)). Passing them via `npm run dev -- ...` only re-introduced the PowerShell `--` consumption bug.
- The PowerShell `Tee-Object` wrapper around npm. Caused UTF-16 LE log files (Tee's default encoding on PS 5.1) which rendered as `c c c` (each char followed by a null byte that renders as a space) when read by the hub as UTF-8.

### Self-healing for existing sessions

`ensure_mandatory_structure` calls `ensure_build_manifests` on every Run Converted click. `ensure_build_manifests` re-emits split run-scripts when `run.bat`/`run.sh` are missing. So:

- New sessions: get the new template immediately when conversion completes.
- Existing sessions: delete `converted/run.bat` (or wait for the broken one to clearly fail), click Run Converted again, scaffold re-emits the new template.

Also during this turn, manually regenerated session `378d02ddd74c`'s `run.bat`/`run.sh`/`_run_backend.cmd` from the new template so the user's mid-debug iteration picks up the fix without having to delete + rerun.

## fix(run,scaffold,review): visible terminal windows + source files render on review page

Two follow-ups after the user asked "why no new command prompt is opened for run converted???" and "the source files which are written are not rendering in review & approval page when clicked on any file of converted file from any folder??"

### Root cause 1 — `start "Title" cmd /k` silently no-ops without an attached console

Confirmed via the April c9afaa02f85e session's hub run log at `logs/runs/c9afaa02f85e/20260526-035611-01141eb747bc.log`:

```text
Both servers are running in separate windows. Close those windows to stop.
[detached] launcher exited rc=0 but http://localhost:5050/ answered 200;
   server is running in detached child windows
```

So `cmd.exe`'s `start "Title" cmd /k ...` CAN allocate new console windows when spawned via subprocess.Popen — but only if the parent process has a console attached. The April hub spawn that "just worked" must have had one. Today's hub spawn does NOT (uvicorn → asyncio.create_subprocess_exec inherits "no console"), so the `start` calls in `run.bat` silently fell back to inline execution, blocking on `cmd /k`'s "wait for input" loop.

### Fix 1 — `backend/agents/run_manager.py` add `creationflags=CREATE_NEW_CONSOLE` (Windows)

Both spawn sites (initial run at ~line 899 + rerun-path at ~line 1173) now pass `creationflags=subprocess.CREATE_NEW_CONSOLE` on Windows. The parent run.bat gets its own console; its `start "Title" cmd /k ...` calls can now legitimately allocate visible backend + frontend windows. The parent's stdout/stderr still go to pipes (the hub run-manager still streams them), so the parent console stays effectively empty — but it exists, which is all that matters for `start` to work.

Conditional on `os.name == "nt"`. Linux/macOS get no creationflag (the kwarg doesn't exist there).

### Fix 2 — `backend/agents/scaffold.py` template uses `start "Title" cmd /k` for both stacks

Replaced the previous `start /B "" _run_backend.cmd` (silent background) + foreground `npm run dev` with:

```bat
start "AppNova Backend"  cmd /k "%~dp0_run_backend.cmd"
... TCP poll ...
start "AppNova Frontend" cmd /k "cd /d ""%~dp0frontend"" && if not exist node_modules ( npm install ) && npm run dev"
```

Both child windows are titled, persistent (`/k` keeps them open after the process exits so the user can scroll back through any startup exception). Parent run.bat then `goto :eof` — exits immediately. The run-manager's `[detached]` heuristic (line ~1450 in run_manager.py) picks up "launcher exited rc=0 + port answers" and marks the run as `running_detached` rather than `crashed`.

### Root cause 2 — source files not rendering on review page

The review page's source-pane fetched `/api/sessions/<sid>/source-file?path=app/Http/routes.php` and got "Source unreadable — file_missing (...wrapper-folder detection likely failed)". The detection logic is `_detect_project_root(source_root)` at [backend/main.py:2765](backend/main.py#L2765): it returns the inner dir IF and only IF `source/` contains exactly one top-level non-skipped entry.

`uploads/<sid>/source/` in current sessions has THREE entries:

```text
TotalBookingAI-Input/   ← legacy upload (the real wrapper)
chat/                   ← AppNova-runtime: per-card chat tree storage
context/                ← AppNova-runtime: file_map.json + deploy_config.json (already skipped)
```

`context/` is in `_PROJECT_ROOT_SKIP_NAMES`, so it's filtered out. `chat/` is NOT, so the post-filter list has `[TotalBookingAI-Input/, chat/]` = 2 entries, the `len(entries) == 1` check fails, and the function returns `source/` itself. Then any source-file lookup like `app/Http/routes.php` resolves to `source/app/Http/routes.php` — which doesn't exist (it's at `source/TotalBookingAI-Input/app/Http/routes.php`) → "file_missing".

The April session c9afaa02f85e didn't have this bug because its `source/` only had `TotalBookingAI-Input/` and `context/` (no `chat/` because no per-card chat had been opened on that session yet). So the detection picked up the wrapper correctly there. Newer sessions broke it when the chat runtime started writing under `source/`.

### Fix 3 — `backend/main.py:_PROJECT_ROOT_SKIP_NAMES`

Added the AppNova-runtime sidecar folders that can land under `source/` after the initial upload:

```python
_PROJECT_ROOT_SKIP_NAMES = frozenset({
    "__MACOSX",
    "context",       # context/file_map.json + deploy_config.json
    "chat",          # per-card chat tree storage that propagates under source/
    "dev-chat",      # dev-chat transcript dir (post-conversion edit loop)
    "screenshots",   # browser-test PNGs
    "attachments",   # chat upload attachments
    ".git", ".github", ".gitlab",
    ".idea", ".vscode",
    "node_modules", "__pycache__",
    "dist", "build", "target",
})
```

REPL-verified: after this change, `_detect_project_root` on all three current sessions resolves correctly to `source/TotalBookingAI-Input/`, and `app/Http/routes.php` exists at the resolved path. The review-page source-pane will now render the legacy file's content side-by-side with the converted target.
