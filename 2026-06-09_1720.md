# 2026-06-09 17:20 — Surface detached backend.log into the run + dev-chat fix loop

## Context
Clicking **Run Converted** on session `58dd9749df6d` opened the backend terminal
and started `dotnet run`, but the run card showed only a generic
`[process_crashed] exited 1`. The actual cause — C# compile errors (CS1003,
CS1061, …) in the converted Aries.API project — was written to
`<converted>/backend.log` by the detached `_run_backend.cmd` window and never
reached AppNova.

## Root cause
The dual-stack `run.bat` launches the backend (and frontend) in **separate**
console windows via `start "..." cmd /k`, redirecting their stdout/stderr to
`backend.log` / `frontend.log`. The run manager only pumped `run.bat`'s own
piped stdio — which contains just the `[AppNova] backend on …` echo lines — so
`run.log` / `run.error_buffer` never captured the children's output. The
dev-chat / browser-test fix loop reads `run.error_buffer` (`get_recent_errors`),
so it was **blind to backend compile errors** and had nothing to fix.

## Change (per user decision: wire backend.log into the loop)
`backend/agents/tools/run_manager.py`:
- New `_tail_log_file_into_run(run, path, stream_label)` coroutine — polls a
  detached child's log file from offset 0, feeds every new line through
  `_log_line` (so error-shaped lines land in `run.error_buffer`) and broadcasts
  it on the SSE stream. Handles run.bat's `del + recreate` truncation by
  rewinding to 0 when the file shrinks. Self-exits on terminal phase after a
  final drain.
- `_launch_and_await_ready` now starts tailers for `backend.log` and
  `frontend.log` (in `run.cwd`) whenever the launcher is a script
  (`run.bat`/`.sh`/`.ps1`). Started **before** the readiness await so an
  early-failing build is still captured. Tasks stored on `run._log_tail_tasks`
  so they aren't GC'd.
- `stop_run` cancels `run._log_tail_tasks` for tidy shutdown (they also
  self-exit on the phase flip).

`README.md`:
- Step 9 ("Run converted") now notes that detached `backend.log`/`frontend.log`
  are tailed into the run's log + error buffers, so a non-compiling converted
  backend surfaces its real errors instead of a generic `process_crashed`.

## Not done (by design)
- Did **not** hand-fix the 148 C# compile errors in the converted output. The
  3 `__DB_ADMIN_USER__` placeholder fixes that were trialled were reverted so
  the post-run dev-chat / browser-test fix loop owns the converted code
  end-to-end — which is now possible because the errors reach `error_buffer`.
