## 1. Claude Code CLI — shared, machine-wide install (2026-05-15)

**Symptom:** `code-analysis` died with
`Claude Code CLI not found at 'C:\Users\Chaitanya\AppData\Roaming\npm\claude.cmd'`.
The `.cmd` shim existed but the npm package payload (`cli.js`) was gone —
a half-broken global install.

**Root cause #2 (found while fixing #1):** Claude Code 2.1.x changed its
package layout. There is no `cli.js` anymore — the entry point is a native
`bin\claude.exe`, with only `cli-wrapper.cjs` / `install.cjs` at the
package root. `runner._resolve_executable` was hard-coded to the legacy
`node.exe <cli.js>` invocation and would have failed even on a clean
reinstall.

**Fixes (committed in `f599fba`):**

- [backend/agents/runner.py](backend/agents/runner.py) `_resolve_executable`
  now tries, in order: (1) `node_modules/@anthropic-ai/claude-code/bin/claude.exe`
  (2.1+ native binary — direct exec, no node wrapper), (2) legacy
  `cli.js` + `node.exe`, (3) clear `FileNotFoundError` naming both
  attempts and the reinstall command. Bypassing the `.cmd` shim is still
  required on Windows because `cmd.exe /c` mangles newline args.
- Installed `@anthropic-ai/claude-code@2.1.142` to a shared public prefix
  `C:\Users\Public\npm-global\` (writable by every Windows user on the
  host, persists across accounts, no admin). The current user's original
  npm `prefix` was restored afterward so their own global installs are
  unaffected.
- [.env](.env) `CLAUDE_CODE_PATH=C:\Users\Public\npm-global\claude.cmd`
  so AppNova resolves the CLI regardless of which Windows user runs it or
  what their PATH looks like.

**Still per-user:** `claude login`'s OAuth token binds to the OS user
under their appdata. Other Windows users must run `claude login` once
themselves — Anthropic's CLI has no shared-machine token store.

## 3. RAG persistent-store path bug (2026-05-15, committed in `f599fba`)

**Symptom:** log line `[RAG] Chroma persistent client ready at .` — the
store was being created in the process cwd instead of `<repo>/chroma`.

**Root cause:** `rag_root_dir()` did `Path(os.environ.get("APPNOVA_RAG_DIR")
or "")` then `if not str(root)`. `str(Path(""))` is `"."`, not `""`, so
`not "."` is `False` — the repo-root fallback never fired.

**Fix:** [backend/playbooks/rag/client.py](backend/playbooks/rag/client.py)
`rag_root_dir()` now checks the env *string* before constructing a Path;
empty/unset → `Path(__file__).resolve().parents[3] / "chroma"`. Verified
it resolves to `C:\Users\Public\AppNovaAI\AppNovaAI-main\chroma`.

## 5. Git hygiene — untrack the accidentally-committed Chroma DB (2026-05-18)

**Problem:** commit `f599fba` swept the Chroma runtime store into version
control — `chroma.sqlite3` (~26 MB) plus a per-collection GUID dir
`806e622a-…/` of `*.bin` files. They were created when the store root
resolved to the repo root (the bug fixed in §3) and got picked up by
`git add`. `chroma` was not in `.gitignore`, so it would recur.

**Fix:**

- [.gitignore](.gitignore): added `chroma/`, `chroma.sqlite3`,
  `chroma-*/`, and a GUID-dir glob
  (`[0-9a-f]{8}-[0-9a-f]{4}-*/`) for the per-collection segment dirs.
- `git rm -r --cached chroma.sqlite3 806e622a-…/` — removed the 5 blobs
  from tracking, kept them on disk (the live store still works; it's just
  no longer versioned). They will not re-enter the index because of the
  new ignore rules.
- The store is fully rebuildable regardless: `__curated` via
  `POST /api/rag/seed`, `__source` re-ingested at the top of every run,
  `__learned` re-accrues as conversions pass the parity gate.

> Note: this only removes the blobs going forward. The ~26 MB still lives
> in history at `f599fba`. If repo size matters, a follow-up
> `git filter-repo --invert-paths --path chroma.sqlite3 --path
> 806e622a-9fc4-49c8-aa8d-b822f9a62ed9` would purge it — left out here
> because history rewrites need explicit sign-off (force-push, shared
> clones invalidated).
