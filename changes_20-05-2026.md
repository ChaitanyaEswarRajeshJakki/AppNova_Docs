## 7. Auto-reattach on 409 — workspace no longer stays blank after Run All

**Problem:** clicking **Run All Agents** on a session whose backend run
was already in flight produced *"Couldn't start: Analysis already running
for this session."* and a blank workspace. The local `state.running` flag
was `false` (this page-load didn't own the original SSE stream — common
after navigating back to the workspace, opening a second tab, or after
the original tab was closed), so the click went through, the backend
returned 409 from the duplicate-run guard at
[backend/main.py:2952](backend/main.py#L2952), and the frontend just
showed the alert. The only recovery was a hard refresh so
`DOMContentLoaded` could fire `reattachIfRunning()`.

**Fix:** in `startAnalysis()`
([frontend/app.js:1500-1524](frontend/app.js#L1500-L1524)), when the POST
returns 409, swallow the alert and call `reattachIfRunning()` instead.
That repaints cards from `/api/session/:id/status` (in-progress list) and
starts the 3 s `/api/results` poller, so the user lands on a live view
without touching reload.

**Why:** the 409 is *informational*, not an error from the user's
perspective — their intent ("show me this run") is identical to what
reattach already does. The hard-refresh dance was pure friction.

**How to apply:** any HTTP 409 from `/api/analyze`, `/api/run-selected`,
or `/api/resume` now triggers reattach. Other non-2xx codes still alert.

## 8. Preamble stripper — three new cues to catch security-agent chatter

**Problem:** card bodies were showing the model's meta-narration before
the actual report — e.g. *"I have gathered all the necessary context.
Now composing the full security report."* `stripReportPreamble()`
([frontend/app.js:2556](frontend/app.js#L2556)) is designed to drop
exactly this kind of intro, but its regex list `_PREAMBLE_CUES`
([frontend/app.js:2546](frontend/app.js#L2546)) missed the bare-verb
*"I have gathered…"* (no leading *now*) and *"Now composing…"* (no
leading *let me / I'll*) variants.

**Fix:** added three regexes to `_PREAMBLE_CUES`:

- `/\bi have (?:gathered|collected|reviewed|analyzed|read|examined)\b/i`
- `/\bnow (?:composing|writing|preparing|drafting|producing|generating)\b/i`
- `/\b(?:composing|writing|…) (?:the )?(?:full \|final \|complete )?(?:report|analysis|…)\b/i`

The existing cut-point logic — first blank line OR first `#` heading
within 12 lines — is unchanged, so the strip remains conservative (no
strip if no cue lands in the first 5 non-empty lines, no chop if no
cut-point appears within 12 lines).

## 11. RAG chunker — never drop oversized symbols, window them instead

**Problem:** [backend/playbooks/rag/chunker.py:211](backend/playbooks/rag/chunker.py#L211)
silently skipped symbols whose body exceeded `_MAX_CHUNK_CHARS` (12000
chars / ~3000 tokens) via `continue`. The whole-file fallback at line
225 only fired when **zero** chunks emerged from a file — so a file
with 5 small functions + 1 god-class would lose the god-class entirely
without warning. For migration-planner / code-generation, that's
exactly the content most likely to carry the legacy app's UI/UX rules.

**Fix:** [backend/playbooks/rag/chunker.py:208-244](backend/playbooks/rag/chunker.py#L208-L244)
— when a symbol exceeds `_MAX_CHUNK_CHARS`, it is now sliced into
windows of `_MAX_CHUNK_CHARS` each (`symbol_kind="{original}_window"`,
`symbol_name="{name}#{win_idx}"`, anchor line adjusted per slice) rather
than dropped. This guarantees every parseable symbol contributes at
least one chunk, satisfying the "no file skipped" contract for
downstream retrieval.

**What this does NOT change:**

- `walk_and_chunk()` skip set (`node_modules`, `vendor`, `.git`,
  `__pycache__`, `_appnova_legacy_runtime`) — these never contain user
  source; correctly excluded.
- `_EXT_LANG` extension allowlist — your current AngularJS/Blade/CSS
  source tree is fully covered. Adding `.cshtml` / `.razor` / `.xml`
  / `.config` is a follow-up if a future project needs them.
- `max_files=5000` cap — your project has 340 files; nowhere near
  the ceiling.
- `_WHOLE_FILE_CEILING=24000` and the whole-file window path — already
  handled the no-symbols-detected case correctly; left alone.

**Acceptance test (manual, when you next run a conversion):** scan the
session's Chroma collection for chunks whose `symbol_kind` ends in
`_window`. Any present means the chunker successfully recovered a big
symbol that would have been dropped on the old code path.

## 14. Run-All-Agents: paint placeholder card immediately on click

warm start, longer on a cold one). Users assumed the click did nothing

## 24. SSE stream-drop hardening — fix "stuck on Discovery card, refresh required"

**Symptom reported by user:**

After uploading a project and clicking Run, the browser console showed:

```
POST /api/run-selected/{sid} net::ERR_INCOMPLETE_CHUNKED_ENCODING 200 (OK)
Stream error: TypeError: network error
POST /api/resume/{sid} 409 (Conflict) — "Nothing to resume — already done"
```

The backend run actually completed successfully (every applicable agent
finished, hence the 409 from resume), but the frontend stayed frozen on
the Project Discovery card. The per-agent cards only appeared after a
manual page reload. The user also asked for live LLM narration ("read
context… Let me create a comprehensive database migration plan for the
TotalBookingAI system…") inside each card while it runs.

**Root cause — single bug, two symptoms.**

The frontend has one SSE consumer ([frontend/app.js:1599-1610](frontend/app.js#L1599-L1610))
that `await reader.read()`s the chunked response from
`/api/analyze | /resume | /run-selected`. When that connection severs
mid-run (the `ERR_INCOMPLETE_CHUNKED_ENCODING` above), three failures
chain:

1. `reader.read()` throws `TypeError: network error`.
2. The catch at app.js:1611 swallows the error with a `console.error`
   and falls through.
3. The fallback `/api/results` fetch at app.js:1632 only renders FINAL
   reports via `renderFinalReport`, which early-returns at
   [app.js:2620](frontend/app.js#L2620) if no per-agent card exists
   yet (`if (!c) return;`). And cards only exist when `agent_start`
   events arrived — which they didn't, because the stream died.

So after the stream drops, the UI is permanently stuck on the Discovery
card until reload, at which point `reattachIfRunning` runs the proper
recovery (which the in-page catch never invoked).

The "no live narration" symptom is the same bug from a different angle:
the per-card streaming text is fed by `agent_event` payloads on the
exact same stream, so when the stream dies, narration stops.

**Why the stream drops on localhost** (no proxy, no nginx, no
Cloudflare):

- Long idle gaps (~tens of seconds between discovery completion and the
  first agent's first tool call) can be killed by Windows TCP / AV /
  endpoint-protection drivers (Defender Application Guard, Symantec,
  etc.) that occasionally reap idle chunked HTTP/1.1 connections.
- One un-encodable event inside the SSE generator would `json.dumps`-
  throw and Starlette would close the chunked response with no trailer
  — exactly the symptom observed.
- `claude-code` subprocess stream-json lines occasionally carry payloads
  that the strict json encoder refuses (Decimal, datetime, bytes leaked
  via a tool result).

**Fix — three layers, fully implemented in this commit:**

### 24a. Frontend auto-reattach on SSE drop ([frontend/app.js:1611](frontend/app.js#L1611))

Replaced the silent
`catch (e) { console.error('Stream error:', e); }` with a call to
`reattachIfRunning({ source: 'sse-drop' })` and an early `return`.
When the stream drops, the frontend now runs the exact same recovery
that today only fires on page reload: paint cards from
`/api/session/{sid}/status` + `/api/results`, then start the 3s
result-poller, then open a live-tail (24c below) so future events
flow in without a refresh.

Also added a `cleanExit` flag inside the reader loop — the loop is
supposed to exit ONLY via `done === true` (server sent `stream_end`).
Anything else now throws into the catch, triggering recovery instead
of falling through silently.

### 24b. Backend SSE generator hardening ([backend/main.py](backend/main.py))

Replaced all five inline `event_stream()` generators (`/api/analyze`,
`/api/resume`, `/api/run-selected`, `/api/run-mode/{sid}`,
`/api/task-planner/apply/{sid}`) with a shared helper:

```python
async def _stream_events_with_keepalive(queue, session_id):
    while True:
        try:
            event = await asyncio.wait_for(queue.get(), timeout=15.0)
        except asyncio.TimeoutError:
            yield ": keepalive\n\n"  # SSE comment — ignored by client
            continue
        if event is None:
            yield f"data: {_safe_dumps({'type': 'stream_end'})}\n\n"
            return
        yield f"data: {_safe_dumps(event)}\n\n"
```

Two protections:

- **15s idle keepalive** — emits an SSE comment line so any idle-
  timeout enforcer (Windows TCP / AV driver / future reverse-proxy)
  sees periodic traffic and never reaps the connection.
- **`_safe_dumps()`** wraps `json.dumps` with `ensure_ascii=False,
  default=str` plus a try/except that substitutes a synthetic
  `encode_error` event on failure. One pathological event from
  `claude-code` no longer kills the whole stream.

### 24c. Per-session SSE event ring buffer + replay-and-tail endpoint

New module state ([backend/main.py:301-318](backend/main.py)):

```python
_SESSION_EVENT_LOG_CAP = 2000
_session_event_log: dict[str, list[dict]] = {}
_session_event_seq: dict[str, int] = {}
_session_event_cond: dict[str, asyncio.Condition] = {}
```

Every event emitted by `on_progress` (and the finally-block terminal
events: `cancelled`, `error`, `cost_report_*`, `stream_end`) now also
flows through `_publish_event(session_id, event)`, which:

1. Stamps the event with a monotonic `_seq` field.
2. Appends to the per-session ring buffer (bounded at 2000 events; 12-
   agent runs emit ~600-1200 events on a real project).
3. Notifies a per-session `asyncio.Condition` so any long-poller
   waiting on new events wakes up immediately.

New endpoint **`GET /api/session/{sid}/events?since=N`**
([backend/main.py:4677-4762](backend/main.py)) — replays every event
in the buffer with `_seq > N`, then long-polls for new events with
the same 15s keepalive cadence, and terminates cleanly when it sees
a `stream_end` in the buffer. This is the live-tail the frontend
opens on reattach.

Also added `latest_seq` to `/api/session/{sid}/status`
([backend/main.py:4732](backend/main.py)) so the frontend can jump
the tail cursor PAST historical events on a fresh page-load reattach
(the cards are already painted from `/api/results`, so replaying
their narration would produce duplicates).

### 24d. Frontend live-tail wiring ([frontend/app.js:629-686](frontend/app.js#L629))

New helper `startLiveTail(sid, sinceSeq)` opens a fetch-streamed
connection to the new tail endpoint and pipes each event through the
existing `handleEvent` — same code path as the primary POST stream.
Two key details:

- **Cursor policy** chosen by `reattachIfRunning`:
  - `opts.source === 'sse-drop'` → start from `state.lastSseSeq`
    (we know exactly what the UI rendered before the drop, so the
    tail resumes with zero gap).
  - Fresh page-load → start from `Math.max(state.lastSseSeq,
    status.latest_seq)` so historical events that already shaped
    the cards via `/api/results` don't replay.
- **AbortController stashed on `state.tailAbort`** — a second
  reattach (e.g. user navigates away + back twice quickly) cancels
  the previous tail before starting a new one. Without this, two
  concurrent tails would double-append to the same per-card
  `assistantAccum` and produce duplicated narration text.

The primary stream consumer ([frontend/app.js:1596-1622](frontend/app.js#L1596))
also advances `state.lastSseSeq` from `ev._seq` on every handled
event so the cursor is always current.

**User-visible result:**

- SSE drop mid-run → UI continues filling in cards within ~1s, no
  refresh.
- Page reload mid-run → cards rebuild from `/status` + `/results`,
  then live narration of the in-flight agent streams into the
  Running card from this point forward.
- Long silent gaps (mid-discovery on a large project) → keepalive
  comments keep the connection alive, no idle-timeout kills.
- One bad event from claude-code → encode_error event surfaces in
  the stream; the rest of the run continues unaffected.

- [backend/main.py](backend/main.py) — ring buffer infrastructure,
  `_safe_dumps`, `_publish_event`, `_reset_event_log`,
  `_stream_events_with_keepalive`, new `/api/session/{sid}/events`
  endpoint, `latest_seq` in `/status`, terminal events publish to

- [frontend/app.js](frontend/app.js) — `state.lastSseSeq` +
