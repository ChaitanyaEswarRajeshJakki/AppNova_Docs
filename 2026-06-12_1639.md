# 2026-06-12 16:39 — "AppNova never forgets the DTO": behaviour-drift gate

## Goal (user)
Make the pipeline never again ship a converted app that "forgets the DTO" — the
booking-create `BEHAVIOUR-DRIFT` where the create/update handler dropped most
request/DTO fields and saved an almost-empty row.

## Root insight
`// BEHAVIOUR-DRIFT:` is an OFFICIAL prompt convention (`prompts.py:534` tells
the agent to mark deviations with it) — but **nothing audited the markers**, so
the agent could drop functionality, tag it `BEHAVIOUR-DRIFT`, and ship clean.
June (`58dd9749df6d`) had 7+ markers (incl. the mass-assignment one); the
hand-tuned April reference has 0. Perfect deterministic signal.

## New auditor `backend/agents/auditors/behaviour_drift.py`
Scans the converted backend for every `// BEHAVIOUR-DRIFT:` marker and classifies
the **persistence / data-loss** subset:
- **Mass-assignment shortfall** — comment/context mentions `mass-assign` /
  `$request->all` / `fillable` / "known scalar fields" / "fields explicitly", or
  names a partial-mapping helper (`MapFieldsToEntity`/`ApplyFields`/hydrate).
- **No-op relationship sync** — a NO-OP body signal (`Task.CompletedTask`, "not
  implemented", "handled elsewhere") near relationship wording (`pivot`,
  `many-to-many`, `sync`, `attach`, `collection`) — a `Sync*` stub that never
  persists child rows.
Writes `docs/BEHAVIOUR_DRIFT.md` + manifest (`persistence_drift_count`,
`total_drifts`). Validated: JUNE 2 persistence (TotalBookingRepository.cs:566
mass-assignment + :657 pivot-sync no-op) of 8 total; APRIL 0.
