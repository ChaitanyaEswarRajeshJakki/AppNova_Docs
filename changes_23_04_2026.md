## Central Project Hub `[DONE]`

User ask: "Central hub to manage modernization projects — list all workspaces/projects · create new modernization project · show last run status (Success/Failed/Running) · quick actions (Resume, View Workflow, View Reports) · notifications (failed runs, approvals pending)."

### What shipped

- `data/projects.json` under an atomic-rename + RLock + corruption-recovery pattern. Per-project record: `{id, name, owner, session_ids[], primary_session_id, target_stack, last_run_id, last_run_status, last_run_at, archived, notifications[], description}`.
- `backend/projects.py` with full CRUD + hub-specific helpers: `list_all / get / create / update / delete / attach_session / detach_session / mark_run_started / mark_run_finished / add_notification / ack_notification / ack_all_notifications / find_by_session`, plus `derive_status()` that joins persisted records with live `_session_tasks` / `_results` so a mid-run backend restart never surfaces a stale "running" badge.
- One-shot boot-time backfill: `projects_mod.init()` scans `uploads/<sid>/source/` and `demo_sessions/<slug>/` so existing installs don't see an empty hub on first deploy.
- 11 new endpoints in `backend/main.py`:

```text
GET    /api/projects[?include_archived]
POST   /api/projects
GET    /api/projects/{id}
PATCH  /api/projects/{id}
DELETE /api/projects/{id}[?hard=true]
POST   /api/projects/{id}/attach        body:{session_id, make_primary?}
POST   /api/projects/{id}/detach        body:{session_id}
GET    /api/projects/{id}/notifications[?unread_only]
POST   /api/projects/{id}/notifications/{nid}/ack
POST   /api/projects/{id}/notifications/ack-all
POST   /api/projects/backfill
```

- Run pipeline hooked at four points without any rewrite: startup backfill, `/api/upload` (optional `project_id` form field + auto-create), `_run_analysis_stream.run_in_background` start/finish stamps, clone + demo-load inherit the parent project.
- Frontend:
  - `frontend/hub.html` — stats strip (active / running / needs-attention / unread), filter tabs (All / Running / Needs attention / Success / Archived), searchable project grid, notification bell with badge.
  - `frontend/hub.js` — 8 s polling (paused when tab hidden), inline create/rename/archive modals, notification dropdown that navigates into the offending project on click, graceful auth-disabled + token-expired fallbacks.
  - `frontend/login.js` — post-login redirect + skip-on-valid-token both now land on `hub.html`.
  - `frontend/index.html` — "‹ Project Hub" sidebar link with live project-name slot.
  - `frontend/app.js` — URL-handshake (`honorHubHandoff` reads `?session_id=&project_id=` and stashes in localStorage), plus `paintHubProjectName()` to decorate the sidebar link.

## Review & Approval Screen + Run-Mode Presets `[DONE]`

User ask: "Review and Approval Screen — side-by-side diff (old vs new) · commenting system · approve/reject changes · partial approvals." Plus Claude-Code-style shortcuts (`/plan`, `/security-review`, `/clear`, `/rename`) from the command mapping exercise earlier in the day.

### What shipped — Review

- `backend/review.py`: per-session store at `data/reviews/<sid>.json` using the same atomic-rename / RLock / traversal-guard pattern. Dataclasses: `Comment` (line-anchored to source/target/general), `FileReview` (per-file verdict), `SessionReview` (envelope with notes + opened_by). `refresh_from_mappings()` re-syncs from `context/file_map.json` WITHOUT dropping existing decisions/comments when the reviewer's tree gets regenerated mid-review.

  - Click any line number → comments pane slides out anchored to that side+line. Post/resolve/reopen/delete flows all round-trip to the backend.
- Sidebar "✓ Review & approve" button added to `index.html` with a live badge (pending count or `💬N` when all decided but comments remain). Badge refreshes every 15 s while a session is active but not running.
- Hub card gained a "Review" quick-action that deep-links into `review.html?session_id=…&project_id=…`.

## Prompt → Task-Breakdown Planner with Editable Sequence `[DONE]`

User ask: "Prompt input (Natural Language instructions) · Prebuilt templates (Assessment, Migration, Refactor) · Task breakdown preview (auto-generated) · Edit task sequence." This section records what shipped; parked follow-ups land in the Phases C + D `[PLANNED]` section further down.

### What shipped

- `backend/task_planner.py` — deterministic planner + templates. `Template` / `PlannedTask` / `PlanPreview` dataclasses. `BUILTIN_TEMPLATES = {assessment, migration, refactor, custom}` with explicit agent ordering per template (not derived from the registry, so a new agent added to `AGENT_REGISTRY` doesn't silently leak into every template).
- `plan_from_prompt()` resolution order:
  1. `agent_ids_override` (user edited the sequence) → validate as-is.
  2. `template_id` resolves → seed from template.
  3. Keyword regex pass over the prompt.
  4. No signals → default to Assessment template.
  5. Merge required-upstream closure.
  6. Filter to applicable agents.
  7. Stable topological sort by `AgentSpec.upstream` (preserves user order among peers).
- Every decision emits a human-readable warning in the preview response. 12 keyword patterns cover security / testing / ui / porting / data / infra / docs / business-rules / architecture / review / integration.
- `validate_sequence()` is the pre-flight gate for `/apply` — unknown IDs, duplicates, missing required_upstream, and "wrong order" each surface with a path-specific message.
- `backend/session_plans.py` — per-session plan persistence at `data/plans/<sid>.json` (atomic-rename + RLock + traversal guard, same as review.py). `SessionPlan` holds `{prompt, template_id, agent_ids, created_at, updated_at, applied_at, applied_run_id, notes}`.
- 4 new endpoints:

```text
GET  /api/task-planner/templates
POST /api/task-planner/preview          — stateless; optional session_id for applicable filtering
GET  /api/task-planner/plan/{sid}
POST /api/task-planner/plan/{sid}       — upsert; does not run
POST /api/task-planner/apply/{sid}      — validate + persist + write user_prompt.md + SSE run
```

- Prompt injection into agent briefs:
  - `/apply` writes `uploads/<sid>/context/user_prompt.md` with a preface marker (`# User Instructions ... ---`).
  - `backend/agents/prompts.py::build_agent_prompt()` gained a `user_prompt_path` kwarg. When set, emits a `## USER INSTRUCTIONS` block FIRST (before target stack, before context files, before YOUR TASK) so the constraint hits the model before any other directive.
  - `backend/agents/supervisor.py` checks for the file's existence and passes it through — so legacy runs via `/api/analyze` / `/api/run-selected` keep their unchanged prompt shape.
- Frontend:
  - `frontend/index.html` — new collapsible `<details class="plan-panel">` block above the heartbeat bar: template pills, prompt textarea with char counter, task list, add-agent row, warnings slot, Save / Apply & Run action bar.
  - `frontend/app.js` — ~350-line plan-panel controller as the final IIFE. Fetches templates on first interaction. Previews on template pick; debounced 800 ms re-preview on prompt-change only when a template is selected (Custom waits for explicit Generate). HTML5 drag-to-reorder (index-based data transfer, no library). Row removal + add-agent picker. Apply & Run consumes the fetch body stream chunk-by-chunk, parses SSE frames, pipes events into the existing `handleEvent` so the UI renders identically to the Run button.
  - Two `window.dispatchEvent(new CustomEvent('appnova:session-bound'))` calls — on reattach and on fresh upload — so the plan panel hydrates a saved plan automatically.
  - Bugfix: `honorHubHandoff()` was wiping the entire URL search string when it saw session/project params. Now it strips only those two so hints like `?view=plan` reach downstream code untouched.
  - `frontend/style.css` — plan-panel styles (~260 lines) with `-webkit-user-select` siblings added for Safari.
  - `frontend/hub.js` — "Plan" card action that navigates with `?view=plan` so the workspace opens with the panel pre-expanded.

## Planner — Phases C + D: Smart Layer + External Templates `[PLANNED]`

Carried forward from the planner design notes (§9 Rollout + §12 Open questions — now inlined here). Phases A + B shipped in the section above; C + D are the follow-on work that the shipped shape was deliberately designed not to foreclose.

### Context

The deterministic planner that shipped today (keyword regex + topological sort) is the floor — it always works offline and never fails. The ceiling above it is two features that were explicitly deferred:

1. **Phase C — LLM-augmented planner**. Let Claude refine the deterministic plan. Useful when the user's prompt mentions concepts the keyword table can't catch ("this is a three-tier enterprise app with strict audit requirements" → the planner should know to include `security` + `documentation` + `business-rules` even though none of those words appear).
2. **Phase D — customer-authored templates**. Let ops drop JSON files into `data/templates/` to add AppNova-custom workflows without touching Python.

---

### Phase C — Smart layer (LLM-augmented planner) `[PLANNED]`

#### Trigger model

A new env var `APPNOVA_SMART_PLANNER=1` flips the planner into "deterministic first, LLM refines" mode. Default off — the floor keeps working with zero config.

#### Flow

1. `plan_from_prompt()` runs as it does today and produces a deterministic `PlanPreview`.
2. When the flag is on, `/api/task-planner/preview` calls `refine_plan_with_llm(plan, prompt, project_signals)`:
   - Fire one `claude -p` call with a fixed JSON-schema response ({tasks: [{agent_id, rationale, confidence}], add: [], drop: [], reorder: []}).
   - Merge LLM suggestions into the deterministic floor: only ADD agents, never remove a user-chosen row. Reorders are warnings, not automatic.
3. The preview response gains a `llm_suggestions` sibling to `tasks` so the UI can render "Claude suggests: + `integration`, + `testing`" as separate chips the user can accept individually.
4. Hard budget cap: one call per preview, Haiku tier (cheap), 15 s timeout. Fall back to the deterministic result if the LLM call fails or returns bad JSON.

#### Files to touch

- `backend/task_planner.py` — new `refine_plan_with_llm()` function, shelled out to `backend/agents/runner._run_claude_once` with a stripped-down prompt that sees only the agent registry + the user's prompt (no project files — we don't want the LLM burning tokens exploring).
- `backend/main.py` — gate the call on `APPNOVA_SMART_PLANNER`. Add a `POST /api/task-planner/refine` endpoint for an on-demand "Improve plan" button, independent of the preview debounce.
- `frontend/app.js` — render `llm_suggestions` as accept/dismiss chips; add an "Improve plan with Claude" button in the plan panel when the flag is on.

#### Cost model

~5k input tokens (registry + prompt) + 500 output tokens on Haiku = ~$0.002 per refine. The "Improve plan" button is the gate — not every keystroke triggers the LLM, only an explicit click.

#### Open question

The plan doc §12 flagged one spec-level question that still applies: does the user want a **full chat experience** in the plan panel (multi-turn "shorter plan please", "what if I skip testing?"), or a one-shot "improve plan" button? Phase C delivers the button. Chat is Phase C-prime, not on the roadmap yet.

### Phase D — External template extensibility `[PLANNED]`

#### Storage shape

Drop JSON files under `data/templates/<id>.json`, same schema as `BUILTIN_TEMPLATES` entries:

```json
{
  "id": "audit-only",
  "label": "PCI Audit",
  "description": "Read-only compliance audit scoped to PCI-DSS concerns.",
  "default_prompt": "Identify PCI-DSS gaps in this codebase — card storage, logging, crypto, access control.",
  "agent_ids": ["security", "business-rules", "code-review", "documentation"],
  "tags": ["compliance", "read-only", "pci"]
}
```

#### Loader behaviour

- `backend/task_planner.list_templates()` gains a `_load_external_templates()` helper that scans `data/templates/*.json` at import time. Cache for 60 s in-process; re-scan on cache miss.
- Validation on load: id matches `[a-z0-9_-]+`, label + description non-empty, every `agent_id` resolves against `AGENT_REGISTRY`. Invalid templates get logged + dropped (never break the planner).
- External id collisions with builtins → external wins, builtin is logged as shadowed. Lets customers override the shipped Migration template with their own tighter version.

#### Endpoint surface

Zero new endpoints. `GET /api/task-planner/templates` already returns the merged list; the UI picks up new entries for free.

#### Files to touch

- `backend/task_planner.py` — `_load_external_templates()`, `_validate_template_dict()`, cache, shadow-log.
- `scripts/smoke_external_templates.py` — drop a sample template, hit the API, assert it appears in the list and can drive a preview end-to-end.

#### Optional follow-on

An admin endpoint `POST /api/task-planner/templates` to upload + validate a template JSON from the UI. Deferred until an actual ops user asks for it — the file-drop path is enough for internal use.
