# 2026-06-11 20:08 — Fix "missing forms/rendering" in June conversion: permission-seed parity

## Symptom (user)
"The latest converted session doesn't have all the forms / frontend rendering
like before, even though everything is converted." Reported while running the
June conversion `uploads/58dd9749df6d/converted/` (namespace `Aries.API`): both
servers came up healthy (frontend :5050, backend :5051), login as
`admin@aries.test` succeeded — but most forms/pages were unreachable.

## Root cause (verified, not a regression)
The forms WERE generated — June has MORE artifacts than the good April run
(95 frontend `.tsx` vs 83; 26 backend controllers vs 16). They just weren't
*reachable*. The React route guards gate on permission strings:

- `ProtectedRoute` uses `useApp().allows(p)` → `user.permissions.includes(p)`
  (`contexts/AppContext.tsx`). `user.permissions` is the login response's
  flattened `Roles → Permissions → Name` list (`AuthController.MapUser`).
- The frontend checks **18 distinct** permission strings (`edit_total_booking`,
  `edit_pc_dec`, `view_admin_panel`, `manage_roles`, `manage_bail_amounts`,
  `aries_search`, `import_offense_codes`, `view_pc_dec`, `manage_pc_dec`, …).
- `DbSeeder.cs` seeded only **3** (`view_bookings`, `manage_bookings`,
  `manage_users`); admin role adds every DB permission, but only 3 existed.

Net: the demo admin held `manage_users` and nothing else the guards wanted, so
`ProtectedRoute` redirected it off nearly every form back to the dashboard — the
forms "disappeared" despite compiling. (Navbar dropdown, Dashboard launchpad
tiles, and the booking sidebars use the same `useApp().allows()`, so they were
hidden by the same gap. `useFence`/`/api/current_permissions` is unused leftover
— no controller serves it.) `any_permission` is a test-only sentinel, excluded.

## Changes
### `uploads/58dd9749df6d/converted/backend/Aries.API/Data/DbSeeder.cs`
- **Permissions block** now seeds the full union of the 18 frontend-checked
  permission names (kept the original 3). **Upsert by Name** — adds only the
  missing ones — so an already-seeded demo DB picks up the additions on next
  startup with NO wipe.
- **Role block** made idempotent: load (or create) the `Administrator` role with
  `.Include(r => r.Permissions)` and link every permission it doesn't already
  have, on every startup. Previously ran only `if (!Roles.AnyAsync())`, so an
  existing role would never gain new permissions.
- Added `using System.Linq;`.
- Validated: `dotnet build -c Debug` → Build succeeded, 0 errors (one
  pre-existing CS8602 nullable warning in TotalBookingRepository.cs, unrelated).

### `uploads/58dd9749df6d/converted/README.md`
- Fixed the stale demo-login line ("admin / alice / bob / carol — no password")
  → `admin@aries.test` / `Password123!`, noting it now holds the full permission
  set so every module is reachable.

### `backend/agents/prompts.py` (code-generation agent — durable product fix)
- **Guardrail E, new rule 4 — permission parity.** The seeded demo account MUST
  hold every permission the frontend guards check; the seeded permission-name set
  MUST be a superset of the checked set; seeder idempotent by name + admin role
  re-linked on startup.
- **New self-check bullet** — diff frontend permission strings
  (`permission="…"`, `allows('…')`, `can('…')`) against seeded `Permission`
  names; any unmatched frontend permission = a page the demo account can't open.
- `py_compile` clean.

## How the user applies it
1. Stop the running backend (Ctrl+C in the `Aries.API` window).
2. `dotnet run` again — the idempotent seeder adds the 17 missing permissions and
   re-links them to the existing Administrator role (no DB delete needed).
3. **Log out and log back in** as `admin@aries.test` / `Password123!` — the
   frontend caches `user.permissions` from the login response, so a fresh login
   is required to pick up the new permissions. Every form/module is then reachable.
