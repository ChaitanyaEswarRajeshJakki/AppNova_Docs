## fix(chat): raise the chat-turn timeout ceiling to a hard maximum of 4 hours

**What**
- **Run-converted dev-chat** ([backend/agents/chat/dev_assist.py](backend/agents/chat/dev_assist.py#L73-L88)): the per-turn Claude Code subprocess timeout was `DEV_ASSIST_TIMEOUT` defaulting to **300 s (5 min)**. Added `_DEV_ASSIST_TIMEOUT_CAP_SECONDS = 4*60*60`; the effective value is now `min(env_or_default, cap)` with the getenv default set to the cap — so the default is **4 h** and any `DEV_ASSIST_TIMEOUT` override is clamped so it can never exceed 4 h. The caller-passed `timeout_seconds` in `run_turn` is clamped the same way.
- **Review/recovery chat** ([backend/agents/chat/chat.py](backend/agents/chat/chat.py#L54-L61)): `chat_turn(...)`'s `timeout` default was **600 s (10 min)**. Added module constant `_CHAT_TURN_MAX_TIMEOUT = 4*60*60`, used it as the default, and clamp `timeout = min(int(timeout), _CHAT_TURN_MAX_TIMEOUT)` inside the function (governs both interactive report/code chat and the background recovery turns).
- **Error copy** ([backend/main.py:6998](backend/main.py#L6998)): the dev-chat `504` handler hard-coded *"exceeded the 5-minute LLM timeout"*. Replaced with a message that reports the actual configured ceiling (`{h}h{mm}m / {s}s`) read from `dev_assist_mod._DEV_ASSIST_TIMEOUT_SECONDS`, so it never drifts from the real value again.

**Why**
- Code-mode chat turns can legitimately run long (a single turn may `npm install` / `dotnet restore`, build, and iterate before replying). At 5/10 min those turns were killed mid-flight, surfacing as spurious `turn_error` / `504`s. Implemented as an absolute clamped maximum — env/callers may lower it but never raise it above 4 h.

## fix(codegen): add deterministic post-codegen structural validation pass

**What**
- New `backend/agents/auditors/codegen_validators.py` — a deterministic (no-LLM) validator that runs after every code-generation pass and writes `docs/CODEGEN_VALIDATION.md`. Four checks, each targeting a build-blocking defect class:
  1. **`.csproj` layout** — flags a project file that is an ancestor directory of another `.csproj` (SDK glob then compiles the nested project's sources, e.g. test sources swept into the web build).
  2. **namespace == type collision** — a type whose name equals its namespace's leaf segment (CS0118).
  3. **orphan `using`** — `using X.Y.Z;` under one of the project's own namespace roots that no file declares (CS0234); scoped to internal roots so framework/package namespaces are never false-flagged.
  4. **placeholder residue** — deploy-template tokens (`__APP_DISPLAY_NAME__` …) that leaked into C# source as **bare identifiers**; occurrences inside string literals are excluded (those are the quarantine pass's deliberate config sentinels).
- Wired into [backend/agents/orchestration/supervisor.py](backend/agents/orchestration/supervisor.py): import added; a guarded call after the existing deploy-audit block stashes a compact summary on `result["codegen_validation"]`. Advisory only — does not compile the project or downgrade status by itself.

**Why**
- These structural classes are exactly what per-file LLM generation can't see; a cheap deterministic sweep catches them before the output is called done. Placed under `auditors/` to match this repo's layout (the 30-Jun log's `backend/agents/codegen_validators.py` path is from the later snapshot).

**Verification**
- `py_compile` clean. Smoke test on `uploads/converted` (56 `.cs`, 2 `.csproj`) → **0** findings (no false positives on the clean tree). Synthetic-defect tree → all four checks fire; placeholder check flagged the bare `__JWT_SIGNING_KEY__` while correctly ignoring the in-string `__APP_DISPLAY_NAME__`.
